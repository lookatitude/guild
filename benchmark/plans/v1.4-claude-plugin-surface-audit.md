# Claude Code plugin surface — Guild v1.3 vs Claude Code offering vs v1.4 additions

This audit was produced by surveying the Claude Code plugin docs (URLs in
frontmatter) at the time of v1.4 planning. It maps **what Claude Code
offers** against **what Guild already uses (v1.3)** against **what v1.4
adds**, and names what Guild deliberately leaves on the table (with
re-entry triggers).

The audit is the binding shape for F-6's `commands/guild.md`,
`hooks/hooks.json`, status-line, and `AskUserQuestion`/`TaskCreate`
integration. Backend's T3d-backend-platform implements against this doc.

## 1. Hook events — comparison table

Claude Code currently dispatches 12 hook events (verified by surveying
`https://docs.claude.com/en/docs/claude-code/hooks`). The table below
compares Guild's current `hooks/hooks.json` registration against the
full surface and names v1.4's additions.

| Hook event | Claude Code dispatches? | Guild v1.3 registered? | Guild v1.4 status |
|---|---|---|---|
| `SessionStart` | yes | **yes** (`bash hooks/bootstrap.sh`) | unchanged |
| `SessionEnd` | yes | **no** | unchanged (left on the table — re-entry trigger F1 below) |
| `UserPromptSubmit` | yes | **yes** (`check-skill-coverage.sh` + `capture-telemetry.js`) | unchanged |
| `PreToolUse` | yes | **no** | **ADDED in v1.4** — no-op log emitter to F-5 JSONL; falls through gracefully if host doesn't dispatch |
| `PostToolUse` | yes | **yes** (`capture-telemetry.js`, matcher `Agent\|Task\|Write\|Edit\|Bash\|Skill`) | matcher widened to `*` (every tool) — pairs with `PreToolUse` to emit one `tool_call` JSONL event per invocation; see §"Tool-call observability — pre/post pairing" below |
| `Notification` | yes | **no** | unchanged (left on the table — re-entry trigger F2 below) |
| `Stop` | yes | **yes** (`maybe-reflect.js`) | unchanged |
| `SubagentStop` | yes | **yes** (`capture-telemetry.js`) | unchanged |
| `PreCompact` | yes | **no** | **ADDED in v1.4** — context-preservation log emitter; falls through gracefully if host doesn't dispatch |
| `TaskCreated` | yes | **yes** (`agent-team/dist/task-created.js`) | unchanged |
| `TaskCompleted` | yes | **yes** (`agent-team/dist/task-completed.js`) | unchanged |
| `TeammateIdle` | yes | **yes** (`agent-team/dist/teammate-idle.js`) | unchanged |

### Net change in v1.4

- **+2 new hook handlers** (`PreToolUse`, `PreCompact`) — both
  log emitters writing to the F-5 JSONL log via the `hook_event` event
  type. **They register and emit ALWAYS when v1.4 is installed**
  (constant overhead — single `appendFile` per host dispatch, p99 <
  50ms per spec Risk 9). The handlers do NOT gate by `--loops`; the
  audit-log contract (D6) requires capturing host dispatches whether
  loops are on or off, and the F-5 JSONL path is a v1.4-new artifact
  whose presence is operator-opt-in by installing v1.4. Both
  pre-approved per spec autonomy policy ("Adding NEW Claude Code hook
  events beyond `PreToolUse` and `PreCompact` requires confirmation;
  those two are pre-approved").
- **0 removed hook handlers.** SC8's byte-for-byte preservation
  applies to the v1.3 USER-VISIBLE surface (skill output, gate
  prompts, free-text fallback path, status-line absence). The F-5
  JSONL path is new in v1.4 and does not exist for v1.3 consumers, so
  there is no v1.3 baseline for that path to drift from.
- **PostToolUse matcher widened** to include the four new tool names so
  the existing telemetry handler picks up `AskUserQuestion`,
  `TaskCreate`, `TaskUpdate`, `TaskList` invocations once they're added
  to `commands/guild.md`'s `allowed-tools` list.

### Falls-through-gracefully contract

`PreToolUse` and `PreCompact` are both newer hook events than the v1.3
hook surface. Older Claude Code installs may not dispatch them. The
v1.4 handlers must:

1. Be a single-shot script that exits 0 on any input shape.
2. Take no positional arguments and accept the host's current JSON
   payload format on stdin.
3. Append one `hook_event` line to the JSONL log per invocation; on
   any malformed payload, append a `hook_event` with `status: "err"`
   and continue (do NOT abort the host run).
4. Have p99 latency < 50 ms per spec Risk 9 — implemented via
   `node:fs/promises.appendFile` with no other I/O.

If the host never dispatches the event, the handler simply never runs;
no error path. Tests pin both "host dispatches → handler appends event"
and "host doesn't dispatch → no event, no error" cases.

## 1b. Tool-call observability — pre/post pairing

Spec §"F-5 / Event types" (line 190) requires every tool invocation
emit a `tool_call` JSONL event with `command_redacted`,
`result_excerpt_redacted`, `latency_ms`, and `status` fields. Hook
matchers alone don't carry pre/post timing; this section pins the
pairing mechanism backend implements.

### `PreToolUse` (always-fires) and `PostToolUse` (matcher widened to `*`)

| Hook | Matcher | Fires before/after | Captures |
|---|---|---|---|
| `PreToolUse` | `*` (every tool) | Before the tool runs | `tool` (name), `command_redacted` (argv-like; redacted), `pre_ts` (millisecond timestamp) |
| `PostToolUse` | `*` (every tool — widened from v1.3's `Agent\|Task\|Write\|Edit\|Bash\|Skill`) | After the tool returns | `tool` (must match the paired Pre), `result_excerpt_redacted` (redacted), `status` (`ok`/`err`/`n/a`), `post_ts`, derived `latency_ms = post_ts - pre_ts` |

Both hooks share a per-(`run_id`, `lane_id`, `tool`, `pre_ts`)
correlation key so the post-handler can resolve the pre-handler's
captured `command_redacted` and timestamp. The correlation table
lives in-memory in the hook handler script (the `node` process is
re-spawned per dispatch, so the table is keyed on disk via a small
JSONL sidecar at `.guild/runs/<run-id>/logs/tool-call-pre.jsonl` —
appended on Pre, read+removed on Post).

### Single `tool_call` event emitted per pair

The `tool_call` event is emitted **only by the PostToolUse handler**,
after it has joined Pre's captured fields with Post's:

```
PreToolUse fires →
  tool-call-pre.jsonl appends one record
  { run_id, lane_id?, tool, command_redacted, pre_ts }
  No tool_call event emitted yet.

PostToolUse fires →
  tool-call-pre.jsonl is scanned for the matching pre-record
  (same run_id+lane_id+tool, pre_ts < post_ts, oldest unmatched).
  If found: emit one tool_call JSONL event with the joined fields,
            remove the pre-record from tool-call-pre.jsonl.
  If not found: emit a tool_call event with command_redacted absent
                (treat as observability gap; status="ok"; result and
                latency captured from Post alone).
```

The pre/post correlation lookup is bounded: `tool-call-pre.jsonl`
is rotated on the same 10 MiB cadence as `v1.4-events.jsonl` (per
`v1.4-jsonl-schema.md`'s rotation rules). On crash-resume, orphaned
pre-records (no matching Post within 5 minutes wall-clock) are
flushed to the main JSONL with `latency_ms: -1` and a `status:
"err"` flag so the gap surfaces in `summary.md`.

### Tools NOT covered

Tools that bypass the host's hook surface (e.g., direct subprocess
spawns inside an Agent that don't go through Claude Code's tool
dispatch) are not captured by the Pre/Post pair — they are out of
the F-5 audit-log scope by spec §"What this schema does NOT cover".

### Tests pin

- Each tool in the canonical list (`Read`, `Write`, `Edit`, `Grep`,
  `Glob`, `Bash`, `Agent`, `Skill`, `AskUserQuestion`, `TaskCreate`,
  `TaskUpdate`, `TaskList`, plus `WebFetch` for researcher) emits
  exactly one `tool_call` JSONL event per invocation.
- `latency_ms` matches `post_ts - pre_ts` within 1ms.
- Out-of-order Pre/Post (race) → orphaned-pre flush after 5 minutes.
- Crash between Pre and Post → orphaned-pre flush on resume.
- `status` field maps correctly: `Bash`/`Edit`/`Write` → `ok`/`err`;
  `Read`/`Grep`/`Glob`/`AskUserQuestion`/`TaskCreate` → `n/a`.

## 3. Status-line conventions

Claude Code's status-line surface (per `https://code.claude.com/docs/en/statusline`):

- Configured via `~/.claude/settings.json` `statusLine` key with a
  `command` field pointing at an executable script.
- Script reads `input_json` on stdin per Claude Code's documented
  schema. Available fields include (at least):
  `model.display_name`, `cost.total_cost_usd`, `cost.total_duration_ms`,
  `context_window.used_percentage`, `rate_limits.five_hour.*`,
  `rate_limits.seven_day.*`, plus session/cwd metadata.
- Script outputs a single line of plain text to stdout.
- Refresh interval configurable via `refreshInterval` key (default
  ~1s).
- `padding: 0` is the recommended setting for plain-text outputs.

### Guild v1.3

No status line shipped. v1.3 operators see Claude Code's default status
line (model + cost) only.

### Guild v1.4 addition

**`scripts/statusline-guild.sh`** — outputs:
```
phase | round | cap | loops-mode | restart-count
```

The script:
1. Reads `input_json` on stdin per Claude Code convention.
2. Resolves the run-id from the `$GUILD_RUN_ID` environment variable
   (the existing Guild convention; the orchestrator exports
   `GUILD_RUN_ID=<run-id>` before invoking lifecycle phases).
3. Reads `.guild/runs/$GUILD_RUN_ID/counters.json` to derive
   `phase | round | cap | loops-mode | restart-count` for the active
   run.
4. Outputs a single line of plain text to stdout. No JSON; no markup.
5. **`$GUILD_RUN_ID` unset** → output the literal string
   `phase: unknown` (single line, plain text). Do NOT scan for a
   most-recent run dir; do NOT guess. The status line is a passive
   reader; ambiguity surfaces as `phase: unknown` so the operator
   notices and exports the var or starts a run.
6. **`$GUILD_RUN_ID` set but `counters.json` missing** → output the
   literal string `phase: <run-id> (initialising)` until the
   orchestrator writes the first counter update. Tests pin both
   transitions (cold-start → counters.json appears → status line
   shifts).

#### Operator setup

The status line works ONLY when `$GUILD_RUN_ID` is exported in the
operator's shell at run-start. Two paths:

1. **Manual export.** Before invoking `/guild`:

   ```sh
   export GUILD_RUN_ID="run-$(date -u +%Y-%m-%d)-$(slug)"
   /guild --statusline "<brief>"
   ```

   The orchestrator picks up the existing `$GUILD_RUN_ID` and uses
   it for the run dir name.

2. **Auto-export by the `/guild` command's bootstrap.** When the
   orchestrator generates a run-id (the default path when
   `$GUILD_RUN_ID` is unset at invocation time), it writes the
   resolved run-id to `.guild/runs/.current-run-id` (a single-line
   file containing the run-id literal) AND emits a stderr message
   reminding the operator to `export GUILD_RUN_ID=<value>` for the
   status line. Subsequent shells in the same session pick up the
   var; new shells need the export.

The script does NOT read `.guild/runs/.current-run-id` directly —
the env var is the single source of truth. The `.current-run-id`
file is for operator convenience (a one-liner the operator can
`cat` and export); it is not the script's input.

### Opt-in gating

Status-line is opt-in via `--statusline` CLI flag OR `GUILD_STATUSLINE=1`
env var (default off, preserves SC8). When opted in, the operator
manually wires the script into `~/.claude/settings.json`'s `statusLine`
block per Claude Code's documented workflow.

### Agent-team-pane interaction (relevant per active decision)

Per `.guild/wiki/decisions/agent-team-default-when-tmux-available.md`,
agent-team is the user's preferred backend when tmux is available.
The status-line script must work in BOTH modes:

1. **Single-process subagent mode** — the status line shows the
   orchestrator's phase + round + cap. Single counter file at
   `.guild/runs/<run-id>/counters.json`.
2. **Agent-team pane mode** — the status line shows per-pane
   information. The script reads
   `.guild/runs/<run-id>/agent-team/pane-<id>.json` for the pane's
   active lane, then resolves to the lane's counters in the shared
   `counters.json`. Backend implements pane resolution.

Both modes use the same script; mode is detected by environment
(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` indicates agent-team mode).

## 4. `commands/*.md` frontmatter conventions

Per `https://docs.claude.com/en/docs/claude-code/slash-commands`, the
recognised slash-command frontmatter keys are:

- `description` — string, shown in `/help`.
- `argument-hint` — string, hint text appended to the command name in
  the picker.
- `allowed-tools` — comma-separated list, restricts the model's tool
  surface inside this command.
- `model` — optional, pin a model tier for this command (Guild does
  NOT use this; per `operators-control-model-selection.md`, the
  operator chooses the model, not the command).
- `disable-model-invocation` — optional boolean.

### Guild v1.3 conformance

`commands/guild.md` frontmatter:
```yaml
---
name: guild
description: "Top-level Guild entry. Runs the 7-step lifecycle: brainstorm → team-compose → plan → context-assemble → execute-plan → review → verify-done. Per guild-plan.md §13.1."
argument-hint: "[brief]"
allowed-tools: Read, Write, Edit, Grep, Glob, Bash, Agent, Skill
---
```

Conformant. Includes a `name` key which is informational; Claude Code
does not require it (the filename is the command name) but it does no
harm. `description`, `argument-hint`, and `allowed-tools` are present
and well-formed.

### Guild v1.4 changes

- `allowed-tools` — additive update per §2 above.
- `description` — extended to mention loop modes:

  ```yaml
  description: "Top-level Guild entry. Runs the 7-step lifecycle: brainstorm → team-compose → plan → context-assemble → execute-plan → review → verify-done. With --loops=<none|spec|plan|implementation|all> (or comma-list, e.g. --loops=plan,implementation), runs internal adversarial loops between specialists per ADR-009. Per guild-plan.md §13.1."
  ```

- `argument-hint` — extended to surface the new flags:

  ```yaml
  argument-hint: "[brief] [--loops=none|spec|plan|implementation|all|<csv>] [--loop-cap=N] [--auto-approve=none|spec-and-plan|implementation|all]"
  ```

#### `--loops` value set — full grammar

The `--loops` flag accepts:

1. **Single keyword** from the set `none | spec | plan | implementation | all`.
   - `none` (default) — no loops fire; v1.3 user-visible behavior.
   - `spec` — only F-1 (L1 architect↔researcher) fires.
   - `plan` — only F-2 (L2 architect↔security) fires.
   - `implementation` — only F-3 (L3/L4/security-review per lane) fires.
   - `all` — F-1 + F-2 + F-3 all fire (equivalent to the comma-list
     `spec,plan,implementation`).
2. **Comma-separated subset** — any combination of `spec`, `plan`,
   `implementation` separated by commas (no spaces), e.g.
   `--loops=plan,implementation` or `--loops=spec,implementation`.
   - The comma-list form **must not include `none` or `all`** —
     `none` is the absence of loops (a single keyword) and `all` is
     a single keyword shorthand. Mixing either with a comma element
     rejects with exit 2.
   - The CLI parser deduplicates and ignores order:
     `--loops=plan,plan,spec` → `{spec, plan}`; the resolved set is
     stored in `counters.json` for the run's metadata block.
3. **Invalid values** — anything else (typos, the literal `security`,
   future tokens) reject with exit 2 and the literal error
   `--loops must be one of none|spec|plan|implementation|all or a comma-list of {spec,plan,implementation}`.

The `GUILD_LOOPS` env var mirrors the same grammar. CLI overrides env
when both are set. Backend's T3a-backend-config implements the parser;
qa's tests pin every keyword + every legal comma-list permutation +
representative invalid forms.

No new frontmatter keys are added. `model` is deliberately NOT used
(operator-controlled per the active decision). `disable-model-invocation`
is not set (Guild's `/guild` is intended to be a model-invocable command;
restricting that would break the lifecycle's first phase).

## 5. Plugin manifest (`.claude-plugin/plugin.json`) — what changes, what doesn't

### v1.3 baseline

Manifest fields per current `.claude-plugin/plugin.json`: `name`,
`version`, `description`, `author`, plus marketplace fields. No
`minimumClaudeCodeVersion` pin.

### v1.4 changes

- `version` bumps to `1.4.0`.
- `minimumClaudeCodeVersion` is **added IFF AND ONLY IFF** T3d's live
  smoke confirms BOTH:
  (a) older Claude Code versions silently skip
  `PreToolUse`/`PreCompact` (i.e. the host does NOT dispatch the new
  hook events), AND
  (b) that silent-skip produces an operator-visible bug (e.g.,
  the F-5 audit log misses entries the operator expects, or
  `summary.md` regen comes out incomplete in a way the operator
  notices).
  If both conditions hold, the pin is added. If either fails, the
  pin is OMITTED — the falls-through-gracefully contract handles
  older hosts and the audit log simply has fewer entries on those
  hosts. Backend (T3d) makes this evidence-based call during smoke;
  RT6 in ADR-009's Re-entry-trigger section names this as
  evidence-conditional.

### Manifest restructure deferred to v2.0

Per spec Non-goals: "No breaking changes to the v1.3 plugin contract.
Manifest field renames, hook directory restructures, and command
signature changes deferred to v2.0 if needed."

Specific items left on the table (with re-entry triggers below):

- Renaming `hooks/agent-team/dist/*.js` paths to a flatter layout.
- Splitting `commands/guild.md` into per-phase commands.
- Re-namespacing `skills/{core,meta,knowledge,fallback,specialists}/`
  into a v2 hierarchy.

These are recorded in `benchmark/FOLLOWUPS.md` (T2-technical-writer
appends them when migration-guide ships).

## 6. What Guild deliberately leaves on the table — re-entry triggers

Each item below is a Claude Code surface Guild does NOT use today, with
a named end-condition for re-entering it.

### F1 — `SessionEnd` hook

**Why not in v1.4:** redundant with `Stop` for the current lifecycle's
"end-of-session" reflection (`maybe-reflect.js` already runs on `Stop`).
**Re-entry trigger:** if a future use case needs a separate "session
about to terminate" signal distinct from `Stop` (e.g., flushing F-5
log buffers across cross-process writers), wire `SessionEnd` then.

### F2 — `Notification` hook

**Why not in v1.4:** Guild's notifications are surfaced through
`AskUserQuestion` (gated) or stderr; no pull-to-foreground notification
flow exists. **Re-entry trigger:** when a v1.x operator-experience pass
adds OS-level notifications (e.g., "your run is waiting for input"),
this is the hook to wire.

### F3 — `Monitor` tool

**Why not in v1.4:** Guild's lifecycle is phase-sequential; long-running
background monitors haven't appeared as a need. **Re-entry trigger:** if
F-7's harness (or a future evolve-loop) needs to react to a polling
signal (e.g., test-suite completion), wire `Monitor` then.

### F4 — `LSP` tool

**Why not in v1.4:** Guild does not host a language-server today.
**Re-entry trigger:** when a specialist (frontend / backend / mobile)
ships a code-intelligence-aware skill that benefits from cross-file
type-resolution, declare `LSP` in that specialist's lane and wire the
plugin.

### F5 — `EnterWorktree` / `ExitWorktree`

**Why not in v1.4:** the benchmark factory does its own
clone-per-run via `runs/<id>/_workspace/` (ADR-003). Adding worktrees
would duplicate the isolation primitive. **Re-entry trigger:** if a
non-benchmark Guild lane needs concurrent-branch experimentation
(e.g., A/B-testing two implementation strategies in one run), wire
worktrees then.

### F6 — `model` frontmatter on commands

**Why not in v1.4:** operator-controlled model selection per the
`operators-control-model-selection.md` decision. **Re-entry trigger:**
if Anthropic ships a per-command-tier feature operators explicitly
request (e.g., a "deep-research-only-on-this-command" pin), revisit
this decision.

All additions are **additive**. SC8's "byte-for-byte v1.3 equivalence
when `--loops=none`" applies to user-visible behavior (skill outputs,
gate prompts, free-text fallback, status-line absent). The F-5 JSONL
path (`hook_event` always-append + `tool_call` pre/post pairing) is a
v1.4-new artifact the operator opts into by installing v1.4 — it's
file-system only, doesn't affect user-visible behavior, and is
covered separately in SC9 (observability), not SC8 (no regressions).
`--statusline` (default off) gates the status-line script. Loop-skill
bodies are gated by `--loops=*`. SC8 golden fixture pins the
user-visible equivalence; SC9 tool-call coverage tests pin the
always-append behavior.
