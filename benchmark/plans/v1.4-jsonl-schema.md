# v1.4 JSONL log schema (`schema_version: 1`)

This doc is the **stable repo-path schema** read at runtime by the v1.4
JSONL writer. The path is `benchmark/plans/v1.4-jsonl-schema.md`
(per spec §Constraints — `.guild/runs/<id>/logs/schema.md` was rejected
because per-run runtime state cannot host a stable committed artifact).
Any v1.x change either preserves backwards compat (additive only) OR
bumps `schema_version` and adds a migration ADR.

Live log path per run: `.guild/runs/<run-id>/logs/v1.4-events.jsonl`.
Rotated archives: `.guild/runs/<run-id>/logs/archive/v1.4-events.<N>.jsonl.gz`.
The legacy `events.ndjson` from `hooks/capture-telemetry.ts` is
**untouched** — F-5 logs are additive, in a separate file, with a
separate schema.

## Encoding rules (apply to every event)

1. **One event per line.** UTF-8 JSON, single line, terminated by `\n`.
2. **Required envelope fields** (every event, no exceptions):
   - `ts` — ISO-8601 UTC timestamp with millisecond precision, e.g.
     `"2026-04-27T07:35:00.123Z"`.
   - `event` — the event-type string from the table below
     (one of the 12 listed). Schema validator rejects unknown values.
   - `run_id` — the run identifier string, matching the directory name
     under `.guild/runs/`. Per spec, run_id is unique per task.
3. **Optional fields are omitted, never `null`.** Validators treat
   `null` as a schema violation; missing key is the absence signal.
4. **Field order is not significant** — JSON objects are unordered. The
   summary regenerator reads by key, never by position.
5. **No trailing whitespace, no comments, no JSON5.** Strict
   `JSON.parse` round-trip required.
6. **Field-size cap.** Any string field > 4 KiB is truncated to
   exactly 4 KiB and suffixed with the literal sentinel `... [TRUNCATED]`.
   Truncation happens AFTER redaction.
7. **Redaction** (see §"Redaction policy" below): applies to
   `command_redacted`, `result_excerpt_redacted`,
   `payload_excerpt_redacted`, `prompt_excerpt`, `assumption_text`,
   `result` fields whenever they appear. Redaction is the writer's
   responsibility — readers consume already-redacted bytes.

## Event types — exhaustive list (12)

The 12 types below are the complete set for `schema_version: 1`. Adding a
13th type bumps schema_version and triggers a migration ADR (per spec
§Constraints "JSONL schema versioning"). Every type carries the envelope
fields `ts`, `event`, `run_id` PLUS the type-specific fields named.

### 1. `phase_start`

Emitted when the orchestrator enters a lifecycle phase
(`brainstorm`, `team-compose`, `plan`, `context`, `execute`, `review`,
`verify`, `reflect`).

| Field | Type | Required | Description |
|---|---|---|---|
| `ts` | string (ISO-8601) | yes | Envelope. |
| `event` | `"phase_start"` | yes | Envelope. |
| `run_id` | string | yes | Envelope. |
| `phase` | enum (see below) | yes | Phase name from the lifecycle. |

`phase ∈ { "brainstorm", "team-compose", "plan", "context", "execute", "review", "verify", "reflect" }`

Example:

```json
{"ts":"2026-04-27T07:35:00.123Z","event":"phase_start","run_id":"run-2026-04-27-v1.4.0-adversarial-loops","phase":"plan"}
```

### 2. `phase_end`

Emitted when the orchestrator exits a phase. Always paired 1:1 with the
matching `phase_start` for the same `phase`. Pairing is by ordering
within a run, not by an explicit join key.

### 7. `tool_call`

**Emission gating is single-rule** (post-round-2 clarification): emitted for every internal Claude Code tool invocation **whenever the v1.4 hooks are registered AND the `PostToolUse` matcher fires** — that is, on EVERY run after v1.4 installs (regardless of `--loops` setting). The `PreToolUse`/`PostToolUse` always-append behavior IS the observability mechanism per audit. The earlier draft mentioned `GUILD_OBSERVABILITY=full` as an alternative gate; that env var is **removed** — there is no separate observability flag. SC9 is satisfied automatically by the always-append hook pairing.

| Field | Type | Required | Description |
|---|---|---|---|
| `ts` | string | yes | Envelope. |
| `event` | `"tool_call"` | yes | Envelope. |
| `run_id` | string | yes | Envelope. |
| `lane_id` | string | optional | Set when the tool runs inside a dispatched lane; absent for orchestrator-side calls. |
| `tool` | enum | yes | One of the closed enum: `Read` \| `Write` \| `Edit` \| `Grep` \| `Glob` \| `Bash` \| `Agent` \| `Skill` \| `AskUserQuestion` \| `TaskCreate` \| `TaskUpdate` \| `TaskList` \| `WebFetch` \| `WebSearch` \| `NotebookEdit` \| `BashOutput` \| `KillShell`. | `result_excerpt_redacted` | string | yes | Redacted, ≤ 4 KiB. May be empty string for tools that return no output. |
| `tokens_in` | integer | optional | Input tokens, when the runtime exposes them. |
| `tokens_out` | integer | optional | Output tokens, when the runtime exposes them. |

## Redaction policy (writer-side, always-on)

Applied to every field that can carry user-supplied or tool-output bytes:
`command_redacted`, `result_excerpt_redacted`, `payload_excerpt_redacted`,
`prompt_excerpt`, `assumption_text`. Implemented in
`benchmark/src/redact-log.ts` (NEW for v1.4 — distinct from the
argv-only redaction in `benchmark/src/runner.ts §178-189` which is
scoped to subprocess argv).

The five pattern groups, applied in order:

1. **Token-shape regex.** Reuses the prefix list from
   `benchmark/src/runner.ts §178-189`: OAuth tokens, API keys, JWT
   prefixes (e.g., `sk-`, `eyJ`, `Bearer `, `ghp_`, …). Match → replace
   the token-shaped substring with `[REDACTED_TOKEN]`.
2. **Home-dir absolute paths.** Strings matching
   `(~|/Users/[^/]+|/home/[^/]+)/(\.claude|\.codex|\.ssh|\.aws|\.gnupg)/[^\s'"]*`
   → replace the suffix after the sensitive directory with `[REDACTED]`,
   keeping the directory prefix for context (e.g.,
   `<HOME>/.claude/[REDACTED]`).
3. **Key=value secrets.** Lines matching the case-insensitive pattern
   `(password|token|api[_-]?key|secret|authorization|bearer)\s*[:=]\s*\S+`
   → replace the value side (everything after the separator) with
   `[REDACTED]`, keeping the key name.
4. **High-entropy strings.** Strings of length ≥ 20 matching base64-like
   (`[A-Za-z0-9+/=]{20,}`) or hex-like (`[0-9a-fA-F]{20,}`) high-entropy
   patterns AND not whitelisted (whitelist: SHA-1/SHA-256-shaped commit
   hashes when context indicates git, run-id strings prefixed `run-`)
   → replace with `<HIGH_ENTROPY_REDACTED>`.
5. **Length truncation.** After patterns 1-4 run, any field still > 4 KiB
   is truncated to exactly 4 KiB followed by the literal sentinel
   `... [TRUNCATED]`.

**Negative cases that pass through:**
- Ordinary command output without secrets (e.g., `"PASS 186 tests passing"`).
- File paths NOT under sensitive home dirs (e.g., `"/etc/hosts"`,
  `"benchmark/plans/adr-006.md"`).
- Short identifiers and hex-shaped substrings < 20 chars.
- Test names, function names, line numbers, percentages.

Tests pin both positive (each of patterns 1-4 redacts) and negative
(non-secret content passes through) cases — see
`benchmark/tests/redact-log-patterns.test.ts` (T6-security owns this
file's pattern coverage).

## Post-rotation summary regen contract

`scripts/v1.4-summary-md-regen.ts` reads BOTH the live
`v1.4-events.jsonl` AND every archived chunk
`archive/v1.4-events.<N>.jsonl.gz` in chronological order
(rotation-N ascending, then by `ts` within each chunk). Output is
deterministic given the same input set. Decompression on the fly via
Node's `zlib.createGunzip()`.

Contract:

1. **Reader takes the lockfile.** `flock(.lock, LOCK_EX)` → snapshot
   live log into a buffer → `flock(LOCK_UN)`. Archives are read
   read-only without locking (immutable post-rotation).
2. **Ordering rule.** Archive chunks first, in `<N>` ascending order;
   live log last. Within each file, `ts`-sorted. (Append-order is the
   primary key; `ts` is the tie-breaker for same-millisecond events
   from parallel processes.)
3. **Schema validation.** Every line is `JSON.parse`-able and matches
   `schema_version: 1` (envelope + type-specific fields). A line failing
   validation is logged to stderr with the line offset and skipped — it
   does not abort the regen. (Schema validator script:
   `scripts/v1.4-log-validator.ts`.)
4. **Output.** Markdown summary at
   `.guild/runs/<run-id>/logs/summary.md`, with sections:
   `## Phases` (one bullet per phase_start/phase_end pair),
   `## Specialist dispatches` (one bullet per dispatch + receipt),
   `## Loop rounds` (table per lane × layer with round counts +
   terminator), `## Gates` (one bullet per gate_decision), `## Tools`
   (aggregate counts per tool, p50/p99 latency), `## Hooks`
   (aggregate counts per hook_name, p50/p99 latency),
   `## Escalations` (one bullet per escalation, with options_offered
   echoed), `## Assumptions` (one bullet per assumption_logged, source
   lane + specialist), `## Codex review` (dev-only block, only present
   when `codex_review_round` events exist).
5. **Determinism.** Two regens against the same JSONL set produce
   byte-identical `summary.md`. Pinned by
   `benchmark/tests/summary-regen-determinism.test.ts`.
6. **Backwards-compat.** Future schema versions read newer JSONL with a
   shim that maps unknown `event` types to a `## Unrecognized events`
   section instead of erroring — this preserves audit-trail completeness
   when a v1.5 logger writes against a v1.4 reader.

## `GUILD_LOG_RETENTION` — parsing and behavior

Spec §"Risks → mitigations" line 279 names `GUILD_LOG_RETENTION` as the
operator-side cap-warning knob. This section pins the parser grammar +
the behavior. Backend (T3c-backend-logging) implements against this;
qa pins both happy-path retention parsing and warn-on-cap-exceeded.

### Default

`GUILD_LOG_RETENTION` is **unset by default**, which resolves to the
literal value `unlimited`. Under `unlimited`:

- Live log rotates at 10 MiB to gzipped archives under
  `.guild/runs/<run-id>/logs/archive/v1.4-events.<N>.jsonl.gz`.
- Archives are **never auto-deleted**. Audit-trail completeness is the
  contract; manual operator cleanup is the only path to remove them
  (`rm -rf .guild/runs/<old-run-id>/`).
- No warnings emit. `unlimited` is the explicit-no-cap sentinel.

### Explicit cap grammar

When set, `GUILD_LOG_RETENTION` accepts:

- A positive integer suffixed with `MB` or `GB` (case-insensitive,
  required). Examples: `100MB`, `2GB`, `512mb`.
- The literal `unlimited` (case-insensitive, alias for unset).
- Anything else (no suffix, non-integer, negative) rejects with exit 2
  at logger-init time + the literal error
  `GUILD_LOG_RETENTION must be a positive integer suffixed with MB|GB, or the literal "unlimited"`.

The parser converts to bytes (`MB` × 1024² ; `GB` × 1024³) and stores
the resolved cap in `counters.json` for the run's metadata block.

### Cap-exceeded behavior — warn-not-delete

When the **total on-disk size** of `.guild/runs/<run-id>/logs/`
(live log + every archived `.jsonl.gz`) exceeds the cap:

1. **Emit a stderr warning** with the literal prefix
   `[guild-log-retention] WARNING: cap exceeded — total=<X>, cap=<Y>` —
   on EVERY rotation event after the cap is first crossed (one warn
   per rotation, not per append, to avoid spam).
2. **Append one `assumption_logged` event** to the JSONL log with
   `assumption_text: "Log retention cap exceeded; archives retained
   for audit-trail completeness; operator-side cleanup required."`
3. **Do NOT delete archives.** Audit-trail completeness is the
   contract per spec Risk-3 mitigation. The operator-side cleanup
   path is manual: delete entire run dirs (`rm -rf
   .guild/runs/<old-run-id>/`) — never partial archive deletion,
   which would corrupt summary regen's chronological-merge contract.
4. **Continue rotating normally.** The cap is a soft warn, not a hard
   stop; the audit log keeps growing.

The warn-not-delete behavior is the binding contract for v1.4.
Future versions could add `GUILD_LOG_RETENTION_HARD_DELETE=1` to opt
into auto-deletion of oldest archives — explicitly out of scope for
v1.4 (would conflict with the audit-trail completeness contract).

### Tests pin

- `unset` → resolves to `unlimited`; no warnings; no deletions.
- `100MB` → parses to `104857600` bytes; rotation crosses cap →
  stderr warning + `assumption_logged` event; archives retained.
- `2GB` → parses to `2147483648` bytes; same warn-not-delete path.
- `unlimited` → same as unset.
- Invalid values (`100`, `100kb`, `-1MB`, `abc`) → reject exit 2 with
  the literal error string.
- Warn-once-per-rotation property: 5 rotations after cap-cross →
  exactly 5 stderr warnings + 5 `assumption_logged` entries.

## Cross-platform fallback

On platforms without POSIX `flock` support (Windows), specialists write
to per-lane log files `logs/lane-<id>-events.jsonl` (no shared file, no
race). The summary regenerator reads every `lane-*.jsonl` file plus any
archives, merges by `ts`, and produces the same `summary.md` shape. The
fallback path is gated by `process.platform === "win32"` at logger
init; POSIX hosts always use the shared-live-log + stable-lockfile
path. Tests pin both the POSIX path (4-parallel-process append) and
the Windows fallback (deterministic per-lane merge).

## What this schema does NOT cover

- The legacy `events.ndjson` written by `hooks/capture-telemetry.ts`
  (untouched per spec §"Existing telemetry untouched").
- Non-Guild logs (subprocess stderr, `_subprocess.stdout.log`, etc.) —
  they live in `.guild/runs/<run-id>/artifacts/`, not `logs/`.
- Per-tool internal state (e.g., the actual contents of files Read or
  written by Edit) — only the redacted excerpt is captured.
- Cross-run aggregation — each run is self-contained; aggregation across
  runs is `/guild:stats`'s job (out of scope for v1.4).
