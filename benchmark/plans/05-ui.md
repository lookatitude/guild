# UI reference — surfaces, routing, data flow

This document is the operator-facing reference for the React app under `benchmark/ui/`. It pairs the architectural contract in `p2-ui-architecture.md` with concrete, surface-by-surface guidance an operator can use to navigate the running app and find the file behind any rendered piece. It does not redecide anything from `p2-ui-architecture.md` or `adr-001-runner-ui-boundary.md`; both are accepted and load-bearing.

## How to read this document

If you want to know **how a surface renders**, look at the page-level component named in the table below and follow the route-and-type path. The component tree itself is in `p2-ui-architecture.md §2`; this document does not restate it. Examples used throughout:

- "If you want to know how runs render, look at `RunDetailPage.tsx`. The route is `/runs/:id`. The data shape is `RunDetailResponse` from `benchmark/src/types.ts`."
- "If a chart is missing data points, the page-level component fetches from the route shown below; check the server's stdout log line (one per request, `<METHOD> <PATH> <STATUS> <DURATION_MS>ms`) for that path."

## Surfaces

The app exposes 5 routes corresponding 1:1 to the 5 surfaces in spec §Success P2. Each row links into the architectural contract — the component tree and per-route fetch table live in `p2-ui-architecture.md §2` and `§2.1`; this table is the operator-facing summary. File anchors below name the actual page component shipped in T3-frontend (see `T3-frontend.md §Follow-ups → route: technical-writer (2)`).

| Surface | Route | Page component file | Backend route(s) | Response type(s) |
| --- | --- | --- | --- | --- |
| Runs table | `/` | `benchmark/ui/src/routes/RunsListPage.tsx` | `GET /api/runs` | `RunsListResponse` |
| Run detail | `/runs/:run_id` | `benchmark/ui/src/routes/RunDetailPage.tsx` | `GET /api/runs/:run_id`, `GET /api/runs/:run_id/artifacts/*` | `RunDetailResponse` (joined `RunJson + MetricsJson + Score + EventLine[]`); raw bytes for artifacts |
| Compare view | `/compare?baseline=<set>&candidate=<set>` | `benchmark/ui/src/routes/ComparePage.tsx` | `GET /api/comparisons/:baseline/:candidate` | `Comparison` |
| Trigger panel | `/cases` | `benchmark/ui/src/routes/TriggerPanelPage.tsx` | `GET /api/cases` (`POST /api/runs` returns 501 in P2) | `CasesListResponse` |
| Demo view | `/demo/:case_slug` | `benchmark/ui/src/routes/DemoPage.tsx` | `GET /api/runs?case=<slug>`, `GET /api/runs/:run_id` | `RunsListResponse`, `RunDetailResponse` |

Leaf components live under `benchmark/ui/src/components/` — `StatusPill.tsx`, `StageGraph.tsx`, `ArtifactList.tsx`, `MetricBreakdown.tsx`, `ScoreCard.tsx`, `DeltaChart.tsx`, `DeltaTable.tsx`. The data-fetching primitive is `benchmark/ui/src/hooks/useFetch.ts` (returns `useFetch<T>` and `useFetchText`). Typed URL builders for all five backend routes live in `benchmark/ui/src/lib/api.ts`. Each route's full per-leaf composition is shown in the mermaid `flowchart TD` in `p2-ui-architecture.md §2`; the per-route fetch contract — including the exact `benchmark/src/types.ts` interfaces every page consumes — is `p2-ui-architecture.md §2.1`. For the on-the-wire request/response bodies, see `04-metrics.md §"API endpoints"`.

### Runs table — `/`

`RunsListPage` fetches `GET /api/runs` and renders a sortable table of run summaries. Default sort: most-recent first (per spec §Success P2). Columns: case · plugin ref · status · `guild_score` · started-at. Each row links to `/runs/:run_id`. If the server returns an empty `runs` array, the page renders the shared `EmptyState` chrome (`p2-ui-architecture.md §2`).

If a run shows `status: pass` but no `guild_score`, the run was imported but not scored; the runs-list endpoint hides those (P2 T2 §Assumptions used) — if you see one, the case-loader emitted a warning at compare time (`compare: skipping <run-id> — no score.json` per `02-case-suite.md`).

### Run detail — `/runs/:run_id`

`RunDetailPage` issues a single fetch to `GET /api/runs/:run_id` for the joined response (`RunJson + MetricsJson + Score + EventLine[]`) and renders four leaves: the stage graph (`StageGraph`, driven by `events`), the artifact list (`ArtifactList`, driven by `run.artifacts`), the metric breakdown (`MetricBreakdown`, driven by `metrics`), and the score card (`ScoreCard`, driven by `score`). `MissingArtifactBadge` renders inside `ScoreCard` when a component carries `reason: "missing_artifact" | "timeout" | "errored"` (the zero-reason taxonomy in `04-metrics.md §"Zero-reason taxonomy"`).

When you click an artifact path in `ArtifactList`, the page fetches `GET /api/runs/:run_id/artifacts/<rest>` for the raw bytes. `RawArtifactViewer` uses `useFetchText` for `.md` / `.ndjson` / `.txt` (Content-Type `text/plain`) and lets the browser handle `application/json` or `application/octet-stream` (P2 T2 §Decisions made). Path-traversal hardening is server-side; the UI does not need to sanitize segments.

### Compare view — `/compare`

`ComparePage` reads `baseline` and `candidate` set-ids from URL search params (`useSearchParams`, `p2-ui-architecture.md §3.3`), fetches `GET /api/comparisons/:baseline/:candidate`, and renders the per-component delta table, the delta chart (the chart library is documented in `benchmark/README.md §Charts`), the regression list, the fix list, and the `ExcludedRunsList` driven by the `excluded_runs` field (the determinism filter from `04-metrics.md §"Comparison delta semantics"` R2).

Holding the selection in the URL gives three operator-visible properties at no cost: deep-linking (paste `/compare?baseline=A&candidate=B` into a note), refresh stability, and a deterministic static URL for the deferred public website (ADR-001 §Decision §3). See `p2-ui-architecture.md §3.3` for the full rationale.

### Trigger panel — `/cases`

`TriggerPanelPage` fetches `GET /api/cases` and renders the case list plus a detail panel. The "go" button is disabled in P2 — `POST /api/runs` returns 501 + `DeferredResponse` (`{"reason":"deferred to P3","docs":"benchmark/plans/03-runner.md"}`); the button surfaces the tooltip "P3 — live runs not wired" per `p2-ui-architecture.md §2`. P3 wires the live runner and flips this surface from read-only to interactive.

### Demo view — `/demo/:case_slug`

`DemoPage` is a curated per-case landing — fetches `GET /api/runs?case=<slug>` (the `?case=` filter is server-side; one-line predicate, P2 T2 §Decisions made) for the case's run list, then `GET /api/runs/:run_id` for the headline run. Renders `DemoSummary`, the demo chart, and a reused `ScoreCard`. The same JSON contract powers the deferred public website (`adr-001-runner-ui-boundary.md §Decision §3`); a static export of `/demo/<slug>` is a deterministic file at a deterministic URL.

## State

State is split into three tiers — server-state, local UI state, and cross-route state — fully specified in `p2-ui-architecture.md §3`. Operator-facing summary:

- **Server-state.** A hand-rolled `useFetch<T>(url)` hook backed by `useState + useEffect + AbortController`, returning `{ data, error, status, refetch }`. A sibling `useFetchText(url)` returns `string` for text-bodied artifact responses. No shared cache — each page navigation refetches against `127.0.0.1`. See `p2-ui-architecture.md §3.1`.
- **Local UI state.** Plain `useState`. Sort column on the runs table, selected artifact in the run detail, selected stage in the timeline, selected case in the trigger panel, wrap-vs-no-wrap toggle in the raw artifact viewer. No state escapes the component that owns it. See `p2-ui-architecture.md §3.2`.
- **Cross-route state.** Held in URL search params (`useSearchParams` from react-router v6), only for the compare view's baseline/candidate selection. No React Context as a global store, no Zustand, no Redux. See `p2-ui-architecture.md §3.3`.

Rejected options (Redux, Zustand, Jotai, MobX, TanStack Query, SWR, React Context as a data store) are made explicit in `p2-ui-architecture.md §3.4` so future-me does not re-litigate.

## Data flow

The data-fetching pattern is **no library — see `p2-ui-architecture.md §4`** for the inline justification (the simplicity-first test from the P2 architect bundle resolves to "no library" because `useState + Context + a typed custom hook` covers all 5 surfaces; concrete cost-of-add for TanStack Query is named in §4). `p2-ui-architecture.md §4.1` lists the four conditions that would trigger ADR-002 *Supersedes* in a future phase: (i) mutations land (P3 wires `POST /api/runs` from 501 to live), (ii) live updates land (SSE/WebSocket push), (iii) cross-page caching becomes load-bearing under operator latency, (iv) multi-tab consistency matters.

End-to-end flow for any rendered piece:

1. The page-level component calls `useFetch<ResponseType>(url)` where `url` is one of the 6 routes in `04-metrics.md §"API endpoints"` and `ResponseType` is the corresponding interface from `benchmark/src/types.ts`.
2. The server (`benchmark/src/server.ts`) handles the request — re-reads from disk every time (no in-process cache, ADR-001 §Decision §2), validates against the case-loader / artifact-importer zod schemas where applicable, and returns the typed JSON.
3. The hook resolves to `{ data, error, status }`; the page passes typed props to its leaf components, which are presentation-only.

The server's stdout log emits one line per request (`<METHOD> <PATH> <STATUS> <DURATION_MS>ms`) — useful when debugging an empty UI surface against the actual fetch path.

## Stack

Vite + React + TypeScript (spec §Constraints; user-confirmed default). The frontend lane (T3) ships the Vite scaffold under `benchmark/ui/`. Concrete picks frontend documents in `benchmark/ui/README.md`:

- **Routing.** `react-router-dom` v6 is the architect's non-binding default (`p2-ui-architecture.md §6`). Frontend confirms or swaps in T3.
- **CSS.** CSS Modules + a small set of design tokens (CSS custom properties for color / spacing / type scale) is the architect's non-binding default. Frontend's call.
- **Charts.** `recharts@2.13.3` (T3-frontend confirmed the architect's default — see `benchmark/README.md §Charts` for the full rationale and rejected alternatives). The chart leaves are `DeltaChart.tsx` on the compare view and the per-case demo chart on the demo view; recharts is bundled into its own ~102 KB gz chunk so the rest of the app loads independently of it.
- **Component-test framework.** `vitest` + `@testing-library/react` + `happy-dom`, matching qa's stack for P2.

## Server contract — operator-facing summary

The local read-server boots via `cd benchmark && npm run benchmark -- serve` (P2 T2 §Decisions made). Default port `3055`, override via `BENCHMARK_PORT` env var or `--port <n>` flag. Binds `127.0.0.1` only — no remote access, no auth (`adr-001-runner-ui-boundary.md §Decision §4`). Static fallback for `benchmark/ui/dist/` is wired so a single `npm run benchmark -- serve` serves both the API and the production build of the React app on the same port (P2 T2 §Decisions made — `static fallback`). Dev workflow runs the API server on 3055 and Vite's dev server on a separate port (typically 5173); the dev server proxies `/api/*` to `127.0.0.1:3055`.

Full request/response schemas, body examples, and error envelopes for every route live in `04-metrics.md §"API endpoints"`. The on-the-wire evidence (curl probes against each route, Content-Type assertions, path-traversal probe) is in P2 T2 backend receipt §Evidence.
