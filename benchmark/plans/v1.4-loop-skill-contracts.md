# v1.4 Loop-skill contracts (machine-checkable)

This doc pins the typed input/output shape and termination semantics for
the three new T2 (orchestration) skills introduced by v1.4:

- `guild:loop-clarify` — F-1 architect↔researcher pre-spec loop.
- `guild:loop-plan-review` — F-2 architect↔security plan loop.
- `guild:loop-implement` — F-3 dev↔tester / dev↔qa / dev↔security
  implementation loops with security restart.

Each skill section below is the binding contract. Backend's
implementation in `skills/meta/loop-{clarify,plan-review,implement}/SKILL.md`
must conform; qa's tests assert the contract literally; verify-done greps
this file for the load-bearing literal strings (sentinel, regex set,
escalation copy, restart cap, `loops_applicable` enum).

## Shared termination contract (applies to all three skills)

A loop terminates when the **challenger** (researcher in L1; security
in L2; tester/qa/security in L3/L4/security) emits a handoff containing
the sentinel below as a standalone line, AND the post-sentinel region
of the handoff body passes the malformed-termination regex set. The
contract is verbatim from spec §"Loop termination contract".

### Sentinel — exact literal string

```
## NO MORE QUESTIONS
```

The sentinel must appear:
- On its **own line** (the entire trimmed line equals the sentinel).
- Exactly once per terminating handoff (multiple occurrences =
  malformed_termination — a challenger that emits the sentinel twice
  has not committed cleanly).
- Anywhere in the body, but everything after it is subject to the
  malformed-termination regex set below; everything before it is the
  challenger's substantive analysis (allowed to mention prior questions,
  concerns, etc. as audit trail).

The sentinel is **distinct** from the dev-only Codex `## SATISFIED`
sentinel (different reviewers, different cap policies, different
runtime scope — Codex review is plugin-development-only and never
ships to consumer runs).

### Malformed-termination regex set — applied ONLY to the post-sentinel region

The three regex patterns below run against the substring of the handoff
body AFTER the sentinel line. If ANY of the three matches, the round is
recorded as `loop_round_end.terminated = "malformed_termination"` and
the loop continues with one extra round. Two consecutive
`malformed_termination` events escalate via `AskUserQuestion` with the
standardized 3-option choice (see §"Cap-hit escalation copy" below).

#### Pattern 1 — lines ending in `?` (unresolved questions)

```regex
/^.*\?\s*$/m
```

Multiline mode (`/m`); matches any line whose final non-whitespace
character is `?`. Catches "are you sure?" and similar leftovers.

#### Pattern 2 — bullet lines starting with hard-blocker words

```regex
/^\s*[-*]\s+(blocker|must fix|cannot proceed|MUST|BLOCKING)\b/im
```

Multiline + case-insensitive (`/im`); matches a bullet (`-` or `*`)
followed by one of the listed hard-blocker keywords at a word boundary.
Pins post-sentinel "MUST review the threat model" type leftovers.

#### Pattern 3 — TODO/FIXME/XXX markers (case-sensitive, word-boundary)

```regex
/\b(TODO|FIXME|XXX)\b/
```

Case-sensitive (no `/i`); word-boundary anchored. Catches dev-leftover
TODO markers without false-positiving on words like `Fixmebox` or
`autofixme`.

### Removed-from-spec keyword set (regression guard)

The keyword set `/concern|issue|gap|missing|undefined/i` was REMOVED
from the post-sentinel set after Codex review round 2 — it produced
false positives on legitimate phrasings like "no concerns remain". Any
implementation that re-introduces this set is incorrect; the
post-sentinel scope alone (patterns 1-3 above) catches the genuine
cases.

### Cap-hit escalation copy — exact literal options

When a loop reaches its cap OR a malformed-termination escalates, the
orchestrator dispatches `AskUserQuestion` with the three options as
exact label strings:

- **`force-pass`** — accept the artifact as-is, log unresolved
  questions to `assumptions.md`, proceed to the next phase.
- **`extend-cap`** — extend the cap by N rounds (user supplies N via
  the question's free-text field), continue the loop.
- **`rework`** — abort the current loop, return control to the
  producing skill (`guild:brainstorm` for L1, `guild:plan` for L2, the
  lane's owning specialist for L3/L4/security) with the unresolved
  questions as input.

These three exact strings are the binding contract for every escalation
point: SC1 (F-1 cap), SC2 (F-2 cap), SC3 (F-3 cap, F-3 restart-cap),
malformed-termination ×2, mid-execution decision auto-pass under
`--auto-approve=implementation` or `=all` (auto-pass logs assumption,
does not surface the dialog — but the option set is identical for
consistency). Logs record which one the user chose in the
`escalation.user_choice` field of the JSONL log (per
`v1.4-jsonl-schema.md` event type 11).

### `AskUserQuestion` payload shape (binding contract)

When escalation fires (cap-hit, malformed-termination ×2, or
restart-cap-hit), the orchestrator dispatches `AskUserQuestion` with
exactly this structured shape:

```json
{
  "questions": [
    {
      "question": "<context-specific question text — names the lane/phase + reason>",
      "header": "Loop escalation",
      "multiSelect": false,
      "options": [
        {
          "label": "force-pass",
          "description": "Accept the artifact as-is; log unresolved questions to assumptions.md; proceed."
        },
        {
          "label": "extend-cap",
          "description": "Extend the cap by N rounds (you'll be asked for N)."
        },
        {
          "label": "rework",
          "description": "Abort the current loop; return control to the producing skill with the unresolved questions."
        }
      ]
    }
  ]
}
```

The `label` field of each option carries the exact escalation-copy
literal (`force-pass` / `extend-cap` / `rework`). The `description`
fields are the binding human-readable text — qa pins these strings.

#### Two-step `extend-cap` flow

When the user picks `extend-cap`, the orchestrator follows up with a
second `AskUserQuestion` call asking for `N`:

```json
{
  "questions": [
    {
      "question": "How many additional rounds to extend the cap by? (positive integer ≤ 256)",
      "header": "Loop escalation — extend cap",
      "multiSelect": false,
      "options": [
        { "label": "4",  "description": "+4 rounds (typical small extension)" },
        { "label": "8",  "description": "+8 rounds (medium)" },
        { "label": "16", "description": "+16 rounds (full additional cap)" },
        { "label": "custom", "description": "Provide a different positive integer (≤ 256)" }
      ]
    }
  ]
}
```

`label: custom` triggers a third `AskUserQuestion` call requesting
free-text input parsed as a positive integer ≤ 256 (same constraint
as `--loop-cap`). Invalid free-text re-prompts up to 3 times before
falling back to a default extension of 4 rounds + an
`assumption_logged` event noting the fall-back.

The `extend-cap` follow-up ONLY fires when the user picks
`extend-cap` from the first call. `force-pass` and `rework` are
single-call decisions.

### Backwards-compat — `AskUserQuestion` unavailable

When the host Claude Code runtime does NOT dispatch `AskUserQuestion`
(older versions; non-interactive `claude --print` modes), the
orchestrator falls back to the v1.3 free-text prompt path:

1. Print to stderr the three options as a numbered list followed by
   the `force-pass` / `extend-cap` / `rework` literal labels.
2. Read a single line from stdin.
3. Match the line (trimmed, case-insensitive) against the three labels;
   reject any other input with a re-prompt.
4. Log the choice to `escalation.user_choice` identically to the
   `AskUserQuestion` path.

The fallback is detected at gate-time by trying the
`AskUserQuestion` call and catching the
`tool-not-available` error. Tests pin both branches.

## Skill 1 — `guild:loop-clarify` (F-1, L1: architect ↔ researcher)

### Purpose

Pre-spec adversarial clarification. Architect proposes scope; researcher
fact-checks and surfaces gaps. Wraps `guild:brainstorm` —
**does NOT replace it**. The loop runs BEFORE `guild:brainstorm` writes
the spec; researcher's outputs feed brainstorm's "Assumptions" section.

### Input shape

```typescript
type LoopClarifyInput = {
  brief: string;                    // The user's initial brief, verbatim.
  loops_mode: "spec" | "all";       // The active --loops value (only spec/all activate L1).
  cap: number;                      // Effective cap (CLI/env-resolved; default 16, ≤ 256).
  run_id: string;                   // .guild/runs/<run-id>/ scope.
};
```

### Output shape

```typescript
type LoopClarifyOutput = {
  status: "satisfied" | "cap_hit" | "escalated" | "rework";
  rounds: number;                   // Total rounds executed (1-indexed).
  architect_handoffs: string[];     // Paths to each architect handoff under .guild/runs/<run-id>/handoffs/loop-clarify/.
  researcher_handoffs: string[];    // Paths to each researcher handoff (last one carries the sentinel on success).
  unresolved_questions: string[];   // Empty on satisfied; populated on cap_hit + force-pass + rework.
  assumptions: string[];            // To be appended to spec's Assumptions section.
  next: "guild:brainstorm" | "abort";  // Always "guild:brainstorm" except on rework→abort.
};
```

### Termination

Researcher emits `## NO MORE QUESTIONS` to terminate cleanly. Sentinel +
post-sentinel regex set per §"Shared termination contract" above.

### Restart semantics

**Not applicable.** Restart is L3/L4/security-only (see
`guild:loop-implement`). L1 cap-hit escalates directly via the
3-option choice; no restart concept here.

### Per-lane counter

L1 has **one cap counter** for the whole brainstorm phase. Counter file:
`.guild/runs/<run-id>/counters.json` key `l1_round`. Resets when
control passes to brainstorm-write-spec (i.e., on
`status="satisfied"`).

### Escalation copy

On cap-hit OR two consecutive malformed-terminations, dispatch
`AskUserQuestion` with the three options
`force-pass` / `extend-cap` / `rework` (exact strings — see §"Cap-hit
escalation copy"). User's choice routed:
- `force-pass` → write `unresolved_questions` to `assumptions.md`,
  return `status="satisfied"` (force-pass-as-satisfied), `next: "guild:brainstorm"`.
- `extend-cap` → user supplies N via free-text; cap extended by N;
  loop continues from current round.
- `rework` → return `status="rework"`, `next: "abort"`. Orchestrator
  returns control to the user.

### Backwards-compat fallback (`AskUserQuestion` unavailable)

Free-text stdin prompt with the three labels. See §"Backwards-compat"
above. Same routing for the user's choice.

### Tests pin

- Agree-immediately path (1 round, satisfied).
- 3-round-converge path (2 rounds with researcher questions, 3rd round
  satisfied).
- Cap-hit-escalate path (cap=2, second round still has questions →
  `AskUserQuestion` fires with the exact three labels; user chooses
  `force-pass` → assumption logged; status returned).
- Malformed-termination path (researcher emits sentinel + leftover `?`
  in post-sentinel region → `loop_round_end.terminated = "malformed_termination"`,
  one extra round runs).
- Two-consecutive-malformed-terminations escalates with the same three
  options.
- `AskUserQuestion`-unavailable fallback uses free-text stdin.

## Skill 2 — `guild:loop-plan-review` (F-2, L2: architect ↔ security)

### Purpose

Plan-defect adversarial review. Security raises plan-defect questions
ONLY (security holes, scope-creep, autonomy-policy gaps, contract drift,
untestable success criteria) — **NOT** general code-style suggestions.
Wraps `guild:plan` — **does NOT replace it**. The loop runs AFTER
`guild:plan` writes the plan and BEFORE the user-approval gate.

### Input shape

```typescript
type LoopPlanReviewInput = {
  plan_path: string;                // Repo-relative path to the plan written by guild:plan.
  spec_path: string;                // Repo-relative path to the spec the plan implements.
  loops_mode: "plan" | "all";       // The active --loops value (only plan/all activate L2).
  cap: number;                      // Effective cap (default 16).
  run_id: string;
};
```

### Output shape

```typescript
type LoopPlanReviewOutput = {
  status: "satisfied" | "cap_hit" | "escalated" | "rework";
  rounds: number;
  architect_handoffs: string[];
  security_handoffs: string[];      // Last one carries the sentinel on success.
  dismissed_questions: Array<{ question: string; rationale: string }>;
  // Architect can mark a question "dismissed because X" with rationale recorded
  // in the plan; that does NOT prevent termination — security must independently
  // signal NO MORE QUESTIONS.
  unresolved_questions: string[];
  next: "gate-3-plan-approval" | "abort";
};
```

### Termination

Security emits `## NO MORE QUESTIONS` to terminate cleanly. Architect's
"dismissed because X" markings do NOT terminate the loop — security
must independently emit the sentinel. Same post-sentinel regex set as
L1.

### Restart semantics

**Not applicable.** Same reasoning as L1.

### Per-lane counter

L2 has **one cap counter** for the whole plan phase. Counter file
`.guild/runs/<run-id>/counters.json` key `l2_round`. Resets when
control passes to user-approval-gate (i.e., on `status="satisfied"`).

### Escalation copy

Same three options `force-pass` / `extend-cap` / `rework`, exact
strings, exact routing:
- `force-pass` → unresolved security questions written to
  `assumptions.md`, plan proceeds to Gate 3.
- `extend-cap` → user supplies N; cap extended; loop continues.
- `rework` → return `status="rework"`, `next: "abort"`. Plan returns to
  `guild:plan` for revision (orchestrator reroutes; out of this skill's
  scope).

### Backwards-compat fallback

Same free-text stdin path as L1.

### Tests pin

- Same three paths (agree-immediately, 3-round-converge, cap-hit-escalate).
- Architect's "dismissed because X" path: security still emits sentinel
  cleanly → terminates; security still has questions → loop continues.
- Plan-defect-only filter: security MUST NOT raise code-style
  suggestions (test asserts a code-style-only review terminates with no
  questions in round 1).

## Skill 3 — `guild:loop-implement` (F-3, L3 + L4 + security with restart)

### Purpose

Implementation-phase adversarial loops, three nested layers per lane,
controlled by the lane's `loops_applicable` plan-block field. The lane's
owner (backend / frontend / mobile / devops / qa / etc.) is the
`writer`; the challenger varies per layer:
- **L3** dev↔tester — `qa-property-based-tests` skill is the challenger.
- **L4** owner↔QA — `qa` (full strategy) is the challenger.
- **security-review** — `security` is the challenger; on `findings`,
  triggers a restart from L3.

### `loops_applicable` enum — five valid values

These are the ONLY accepted values for the plan-block `loops_applicable`
field. Plan-validate (T3a-backend-config) rejects others with exit 2 +
the error `loops_applicable must be one of: none, l3-only, l4-only, both, full`.

| Order | Value | L3 runs? | L4 runs? | security-review runs? |
|---|---|---|---|---|
| 1 | `none` | no | no | no |
| 2 | `l3-only` | yes | no | no |
| 3 | `l4-only` | no | yes | no |
| 4 | `both` | yes | yes | no |
| 5 | `full` | yes | yes | yes |

The five values appear in this fixed order in the enum literal.

### Default per lane type (when `loops_applicable` is unset)

| Lane owner | Default `loops_applicable` |
|---|---|
| `backend`, `frontend`, `mobile`, `devops` | `full` |
| `qa` (when primary implementer of test fixtures) | `l4-only` |
| `technical-writer` / `copywriter` / `social-media` (user-facing deliverable) | `l4-only` |
| `researcher` / `architect-as-pure-design` / `marketing` / `sales` / `seo` / non-user-facing copy | `none` |
| `security` (rare — owning an implementation lane) | **plan must explicitly set**; no default. Plan-validate rejects security-owned lanes that omit `loops_applicable`. The T6 review-with-pattern-patch carve-out is the only path to `none` for a security lane (uses the literal comment marker `# review lane; loops_applicable=none per T6 carve-out`). |

#### Why security-owned lanes must explicitly set

A security-owned implementation lane cannot also run the security
review against itself (would be self-review, defeating the adversarial
contract). The plan must EITHER:
- (a) set `loops_applicable: both` and route security-review to a
  different specialist via plan-level override, OR
- (b) set `loops_applicable: l4-only` if no security-review is needed
  for that lane (e.g., a redaction-pattern patch is its own gate),
  OR
- (c) for the **T6 review-with-pattern-patch carve-out** (T6-security
  maintaining `benchmark/src/redact-log.ts` patterns): set
  `loops_applicable: none` with the literal end-of-line comment
  marker `# review lane; loops_applicable=none per T6 carve-out`
  on the same plan-block line.

Plan-validate's full decision tree:

1. Security-owned lane omits `loops_applicable` → reject exit 2 with
   literal error
   `security-owned lane <lane_id> must set loops_applicable explicitly`.
2. Security-owned lane sets `loops_applicable: none` WITH the
   literal comment marker
   `# review lane; loops_applicable=none per T6 carve-out` →
   ACCEPT (T6 exemption).
3. Security-owned lane sets `loops_applicable: none` WITHOUT the
   comment marker → reject exit 2 with literal error
   `security-owned lane <lane_id> sets loops_applicable=none without the T6 exemption marker`.
4. Security-owned lane sets `l3-only`, `l4-only`, `both`, or `full`
   → ACCEPT (normal path).

The comment marker is greppable. Backend's T3a-backend-config
implements the validator; qa pins each of the four cases.

### Input shape

```typescript
type LoopImplementInput = {
  lane_id: string;                  // From plan, e.g., "T3a-backend-config".
  owner: string;                    // Specialist name, e.g., "backend".
  loops_applicable: "none" | "l3-only" | "l4-only" | "both" | "full";
  loops_mode: "implementation" | "all";
  cap: number;                      // Effective cap (default 16).
  restart_cap: number;              // Default 3 per lane per task.
  run_id: string;
  task_id: string;
  prior_receipts?: string[];        // On restart, populated with paths to receipts moved to handoffs/superseded/.
  security_findings?: string[];     // On restart, populated with security's blocking findings verbatim.
};
```

### Output shape

```typescript
type LoopImplementOutput = {
  status: "satisfied" | "cap_hit" | "escalated" | "rework" | "restart_cap_hit";
  layers_run: Array<"L3" | "L4" | "security-review">;
  rounds_per_layer: { L3?: number; L4?: number; "security-review"?: number };
  restart_count: number;            // 0 on first run; increments per security-triggered restart; max 3.
  superseded_receipts: string[];    // Paths under handoffs/superseded/ when restart fires.
  unresolved_questions: string[];
  assumptions: string[];
  next: "next-lane" | "abort";
};
```

### Termination per layer

Each layer terminates independently when its challenger emits
`## NO MORE QUESTIONS` with a clean post-sentinel region:
- L3: tester (`qa-property-based-tests`) emits sentinel.
- L4: qa (full strategy) emits sentinel.
- security-review: security emits sentinel WITHOUT findings; with
  findings, triggers restart.

Same post-sentinel regex set (patterns 1-3) as L1/L2.

### Restart semantics — security restart

When security-review surfaces blocking findings, the restart trigger
is machine-checkable. Security's handoff contains the
`## NO MORE QUESTIONS` sentinel AND a structured findings section
ABOVE the sentinel.

#### Findings section format (machine-checkable)

The security receipt's body contains a heading matching the regex
`/^##\s+(Findings|Open issues|Blockers)\b/im` BEFORE the
`## NO MORE QUESTIONS` sentinel. Under that heading, each finding is
a YAML-style bullet with two required fields:

```
## Findings

- severity: high
  addressed_by_owner: false
  description: <free-text describing the finding>

- severity: medium
  addressed_by_owner: true
  description: <free-text>
```

- `severity` is one of `high | medium | low` (case-sensitive,
  unquoted).
- `addressed_by_owner` is one of `true | false` (case-sensitive,
  unquoted) — the lane owner sets this when they revise their
  handoff in response to the finding (initially `false` when
  security writes the finding).
- `description` is free text; redaction applies before the receipt
  is written to disk.

The `loop-implement` skill parses this section as YAML-bullet syntax
(per-bullet keys, `key: value` pairs).

#### Restart trigger condition

`loop-implement` reads the most recent security receipt for the lane
and parses every finding entry. The restart fires if **any single
finding has `severity: high` AND `addressed_by_owner: false`** —
this is the binding condition. Lower-severity findings, or findings
the lane owner has marked addressed, do NOT trigger a restart on
their own; they are recorded in the lane's audit trail but the loop
proceeds.

Tests pin:
- `severity: high` + `addressed_by_owner: false` → restart fires.
- `severity: high` + `addressed_by_owner: true` → no restart;
  finding logged in `assumption_logged` event.
- `severity: medium`/`low` (any addressed value) → no restart;
  finding logged.
- Findings section absent or no bullets matching the schema → treat
  as no findings; no restart.
- Malformed bullet (missing `severity` or `addressed_by_owner`) →
  log `assumption_logged` event with literal text
  `Malformed security finding bullet — treated as no-restart;
  lane <lane_id>; round <N>` and proceed without restart. (Defends
  against typos blocking the lane forever.)

After processing the findings:

1. **Move prior receipts.** L3 + L4 + security-review receipts for this
   lane move to `.guild/runs/<run-id>/handoffs/superseded/<lane_id>-restart-<N>/`
   (where `<N>` is the restart counter starting at 1).
2. **Cross-reference.** Each prior receipt gains a frontmatter field
   `superseded_by: <new-receipt-path>` (relative path). The
   `superseded_by` cross-reference is the audit trail; future
   `summary.md` regen reads both old and new chains.
3. **Reset L3/L4/security counters for this lane.** Per spec §"Cap
   reset boundaries", a security restart resets all three layers'
   counters to 0 for this lane. This does NOT affect other lanes'
   counters (per-lane isolation).
4. **Increment restart counter.** `counters.json` key
   `<lane_id>.restart_count` incremented by 1. **Restart cap = 3** per
   lane per task — the 4th restart triggers escalation.
5. **New context bundle.** The restarted lane's input bundle includes
   the security findings verbatim, plus pointers to the superseded
   receipts.
6. **Per-lane counter isolation.** Lane A's restart does NOT affect
   lane B's counters. Parallel lanes run independently. Tests pin a
   2-lane-A-restarts-B-continues case.

### Restart cap = 3

The literal restart-cap default is **3** per lane per task. Configurable
only via plan-level override (per-lane), not via a global env var (per
spec — "no global state for restarts"). On restart-cap-hit, escalate to
the user with the standardized 3-option choice
`force-pass` / `extend-cap` / `rework`:
- `force-pass` → log security findings to `assumptions.md`, return
  `status="restart_cap_hit"` with the lane marked
  `force-passed-with-findings`.
- `extend-cap` → user supplies N (additional restart attempts);
  `restart_cap` extended by N; loop re-attempts.
- `rework` → return `status="rework"`. Orchestrator routes to
  user-decision (out of skill scope).

### Per-lane counters — isolation contract

Counters are keyed by `lane_id` in `counters.json`:

```json
{
  "T3a-backend-config": {
    "L3_round": 2,
    "L4_round": 3,
    "security_round": 1,
    "restart_count": 0
  },
  "T3b-backend-loops": {
    "L3_round": 1,
    "L4_round": 1,
    "security_round": 0,
    "restart_count": 0
  }
}
```

Lane A's counter modifications NEVER touch lane B's keys. Tests pin
parallel-lane non-interference under both happy-path and
A-restarts-B-continues scenarios.

### `counters.json` concurrency contract

Specialists run in separate OS PIDs (subagent processes; agent-team
panes). Multiple lanes can update `counters.json` concurrently. The
write protocol:

1. **Atomic-rename.** All writes use the write-tmp + rename pattern:
   write the full updated JSON object to
   `.guild/runs/<run-id>/counters.json.tmp` (`O_WRONLY|O_CREAT|O_TRUNC`),
   `fsync`, then `rename(counters.json.tmp, counters.json)`. POSIX
   `rename` is atomic on the same filesystem; readers always see
   either the pre-update or post-update full file, never a half-written
   one.
2. **Lock reuse.** Concurrent counter-updates serialize on the SAME
   stable lockfile sidecar
   `.guild/runs/<run-id>/logs/.lock` defined in ADR-009 §"Stable-lockfile
   race control architecture". Do NOT add a second lockfile —
   per-run `.lock` is the single coordination primitive for ALL
   per-run shared state (JSONL log + counters.json). The lock is held
   from `read(counters.json)` through `rename(counters.json.tmp, counters.json)`
   and released after.
3. **Optimistic-retry on conflict.** Read-modify-write proceeds:
   read counters → mutate the in-memory copy → write tmp → rename.
   If the read-modify-write fails (rename returns EBUSY/EIO; tmpfile
   write fails on full disk), retry up to 3 times with bounded backoff
   (10ms, 50ms, 200ms). Failure after the 3rd retry → log a
   `tool_call` event with `tool: "Bash"`,
   `command_redacted: "counters.json atomic-rename"`,
   `status: "err"`,
   `result_excerpt_redacted: "<error message>"`, AND propagate the
   error up to the orchestrator. Orchestrator surfaces a user-facing
   message (via `AskUserQuestion` if available, else stderr) naming
   the lane and offering retry/abort.
4. **Crash-resume reads use the same protocol.** On orchestrator
   restart after a crash, the resume path reads `counters.json` under
   the lock; if the file is absent (crash during the very first
   write), counters initialize from zero. If the `.tmp` file exists
   (crash between write-tmp and rename), it is deleted at lock
   acquisition and the read proceeds against the prior `counters.json`.

The read path also takes the lock (read-only `flock(LOCK_SH)` is
acceptable but the v1.4 implementation uses `LOCK_EX` for simplicity
— the contention window is small). Tests pin: 4-parallel-process
counter increment converges to the correct total; rotator-rotator-style
rename race resolves deterministically; orphaned `.tmp` cleaned up on
resume.
