# ADR-009 — v1.4 internal adversarial loops + plugin-surface restructure

## Status

**Accepted** — 2026-04-27 as part of v1.4.0 (`release/v1.4.0`).

This ADR captures the architecture-level decisions for the seven
features F-1 through F-7 of v1.4: internal adversarial loops between
Guild specialists (L1/L2/L3/L4/security with restart), plugin-surface
additions (`PreToolUse`, `PreCompact`, `AskUserQuestion`,
`TaskCreate`/`TaskUpdate`, status line), JSONL log schema versioning,
and the stable-lockfile race-control design.

It is **additive** to ADR-003 (fresh-fixture per run), ADR-004
(process-group signaling), ADR-006 (claude v2.x argv via stdin), and
ADR-008 (interactive harness, design-only). It explicitly does NOT
supersede them; it composes on top. ADR-005 (Windows symmetric of
ADR-004) is referenced as the precedent for "design-only, deferred
implementation" — v1.4 does NOT defer; it ships.

## Context

v1.3 shipped a working Guild lifecycle but with three structural gaps
the user identified and made the v1.4 brief:

1. **No adversarial loop between specialists.** Brainstorm → plan →
   execute were sequential single-pass dispatches. A specialist's
   first-draft output went straight to the next phase. Errors that
   would have been caught by a second specialist's challenge
   (architect's scope ambiguity vs researcher's facts; plan defects
   security would catch; implementation gaps qa or security would
   catch) leaked through to release.
2. **No native UI for gate decisions.** Gates 1, 2, 3 used free-text
   prompts. Claude Code's newer `AskUserQuestion` tool offers a
   structured multiple-choice surface, but Guild was not wired to use
   it. Same for `TaskCreate`/`TaskUpdate` for visible per-round
   tracking.
3. **No comprehensive audit log of internal lifecycle dispatches.**
   `events.ndjson` (from `hooks/capture-telemetry.ts`) captures hook
   dispatches but nothing tool-call-level or loop-round-level. A user
   asking "show me every internal command this run executed" had no
   answer.

The user's instructions on 2026-04-27 pinned three concrete decisions:

1. Loops are **opt-in** via a `--loops` flag at run-start. With
   `--loops` unset, behavior is byte-for-byte identical to v1.3.
2. The default loop cap is **16 rounds** per loop pair. Configurable
   via `--loop-cap` and `GUILD_LOOP_CAP` env mirror.
3. The plugin restructure is **non-breaking only**. Audit + additive
   improvements; no rename of dirs, no manifest field renames.

Codex adversarial review round 1 against the spec surfaced ~60
concerns. Material additions from round 1 (now in the spec) include:
the malformed-termination regex set, the standardized
`force-pass`/`extend-cap`/`rework` escalation copy, the security
restart-cap of 3, the `loops_applicable` per-lane field with five
valid values, `hook_event` as a NEW explicit JSONL event type
(distinct from `tool_call`; both `hook_event` and `codex_review_round`
are net-new in v1.4 — see `v1.4-jsonl-schema.md` for the
authoritative ordinal numbering of all 12 event types), the
stable-lockfile race-control design, and the JSONL schema-versioning
policy.

This ADR is the architect's binding output. Backend (T3a-d) implements
against it; qa (T5) validates against it; technical-writer (T2)
documents from it; security (T6) threat-models its surfaces.

## Drivers

| ID | Driver | Source | Why it matters |
|---|---|---|---|
| **D1** | **Loops must compose with — not replace — existing skills.** | spec §"F-1/F-2/F-3" feature decomposition; user instruction "loops integrate with `guild:brainstorm` and `guild:plan`" | The 14-specialist roster and the 7-phase lifecycle are the v1.3 contract. Replacing `guild:brainstorm` with a "brainstorm-with-loop" skill would be a breaking rename. Wrapping it preserves the contract. |
| **D2** | **Loops must be opt-in; default behavior unchanged.** | user instruction 2026-04-27; spec SC8 golden fixture | v1.3 consumers cannot pay a token cost or behavior shift just because they upgraded. Opt-in via `--loops` flag preserves SC8's byte-for-byte guarantee. |
| **D3** | **Termination must be machine-checkable, sentinel-based, with regex hardening.** | spec §"Loop termination contract"; round-1 Codex review | "Specialist agrees" is too soft for a deterministic gate. A literal sentinel (`## NO MORE QUESTIONS`) with a post-sentinel regex set is greppable and rejection-free. |
| **D4** | **Cap behavior must be predictable, per-phase, persisted.** | spec §"Cap reset boundaries"; user instruction "cap = 16 default" | Operators need to reason about cost ceilings. Caps that reset across phases prevent compounding token cost across a run; persistence across crash/resume preserves correct round counts on retry. |
| **D5** | **Plugin-surface additions must be additive — no breaking changes to v1.3 contract.** | user instruction "non-breaking only"; spec Non-goals "no breaking changes" | New `allowed-tools` entries appended (not removed), new hook events added (existing untouched), new env vars all default to their v1.3-equivalent state. |
| **D6** | **The audit log must capture every internal dispatch with a stable schema.** | spec §"F-5 logging discipline" SC5, SC9 | "Show me every command" is the contract. 12 explicit event types (incl `hook_event` distinct from `tool_call`) cover every dispatch surface. Schema-versioned for forward evolution. |
| **D7** | **Race control must work across processes, not just within one Node event loop.** | spec §"F-5 race control"; agent-team-default decision | Specialists run in separate OS PIDs (subagent processes; tmux panes). An async queue inside one process is insufficient. POSIX `flock` on a stable lockfile is the smallest portable primitive. |
| **D8** | **Loop overhead must be bounded.** | spec Risk 9 (`< 50ms p99` per tool call); ADR-004 §M11 cleanup discipline precedent | `PreToolUse` fires on every tool call; large lanes amplify per-call overhead. The handler must be a single appendFile, no fan-out, no synchronous I/O beyond the lock. |
| **D9** | **Internal Guild loops must be distinct from the dev-only Codex review.** | active decision `codex-adversarial-review-loop`; spec §"Standardized escalation copy" footnote | Different sentinels, different caps, different scopes. Conflating them in a consumer's mind would either over-specify cost (Codex never ships) or under-specify (Codex sentinel mistaken for Guild's). |

## Options considered

The decision space has four layered options because the four artifacts
this ADR produces (loop-skill design, plugin-surface integration,
schema versioning, race control) are coupled. Below, each layer is
weighed independently, then a consolidated decision lands the lot.

### Layer 1 — Loop-skill shape

| Option | Mechanism | Trade-offs |
|---|---|---|
| **A1. Replace `guild:brainstorm` / `guild:plan` with loop-aware versions.** | Rewrite the existing skills to embed adversarial loops directly. | **+** No new skill files. **−** Breaking change to the v1.3 contract (rename effective even if filename preserved). **−** Operators with `--loops` unset still pay the new code path. **−** Couples loop logic into prompt-shaping logic that should be orthogonal. |
| **A2. Wrap with new skills (`guild:loop-clarify`, `guild:loop-plan-review`, `guild:loop-implement`).** | Three new T2 skills under `skills/meta/` that wrap the existing skills, only invoked when `--loops=*` is set. | **+** Existing skills untouched (D5 satisfied). **+** Loop logic isolated to its own skill files. **+** Easy to disable per `--loops` mode. **−** +3 skill files; +1 dispatch hop per phase when active. |
| **A3. Add a single dispatcher skill (`guild:loop`) that branches by phase.** | One skill file, switch-statement on phase. | **+** Single point of edit. **−** A monolithic switch dilutes per-phase contracts; testing one path per phase is harder; per-skill T2 contracts are clearer. |

### Layer 2 — Plugin-surface integration

| Option | Mechanism | Trade-offs |
|---|---|---|
| **B1. Wire all 4 new tools (`AskUserQuestion`, `TaskCreate`, `TaskUpdate`, `TaskList`) and all 2 new hooks unconditionally.** | Every Guild run gets the new surface. | **+** Simplest gating logic. **−** Breaks SC8 byte-for-byte (the visible UI changes); **−** older Claude Code installs may not dispatch the new hooks → fall-through requires runtime detection anyway. |
| **B2. Gate the visible UI (`AskUserQuestion`/`TaskCreate`) by `--loops`; wire hooks unconditionally with no-op handlers.** | Hooks fire always but emit nothing useful when `--loops=none`; UI tools only invoked under `--loops=*`. | **+** SC8 preserved; **+** older hosts skip the new hooks gracefully. **−** A `PreToolUse` no-op on every tool call still costs ~ms (D8). |
| **B3. Hooks always-append when registered (constant overhead); UI tools (`AskUserQuestion`/`TaskCreate`) gated by `--loops`; status line gated by `--statusline`.** | Hooks register and unconditionally append one `hook_event` per host dispatch (single `appendFile`, p99 < 50ms per spec Risk 9 — the operator-opt-in cost of installing v1.4). UI tools only invoked under `--loops=*` (free-text path used otherwise — preserves SC8 user-visible behavior). Status-line script only fires when `--statusline` / `GUILD_STATUSLINE=1`. | **+** SC8 user-visible behavior preserved (gate prompts unchanged under default flags). **+** D6 satisfied (audit log captures hook dispatches whether loops are on or off — F-5 JSONL is a v1.4-new path, no v1.3 baseline to preserve). **+** D8 satisfied — constant per-dispatch overhead, no fan-out. **−** Constant `appendFile` cost on every host hook dispatch when v1.4 is installed (operator pays this whether they use loops or not); justified by the F-5 audit-log contract. **−** Slightly more config surface (3 gates: `--loops`, `--statusline`, `--auto-approve`). |

### Layer 3 — JSONL schema versioning

| Option | Mechanism | Trade-offs |
|---|---|---|
| **C1. Per-run schema doc copy at `.guild/runs/<id>/logs/schema.md`.** | Each run carries its schema version with it. | **+** Self-contained run dirs. **−** `.guild/runs/` is per-run runtime state and CANNOT host stable committed artifacts; consumer-side validation against a moving target. **(Originally drafted; rejected after Codex round 1.)** |
| **C2. Stable repo-path schema doc at `benchmark/plans/v1.4-jsonl-schema.md` with `schema_version: 1`.** | Schema doc committed once; runtime reads it from repo path. | **+** Stable, greppable, audit-trail-clean. **+** Future v1.x additions either preserve compat or bump version + add migration ADR. **−** Requires runtime path resolution to repo root. |
| **C3. Schema embedded in code (`benchmark/src/schemas/v1.4.ts`).** | TypeScript const + JSON Schema literal. | **+** Type-safe at write time. **−** Doc-as-code — humans can't read it without running tsc; ADR review against a code file is harder than against markdown. |

### Layer 4 — Race control across processes

| Option | Mechanism | Trade-offs |
|---|---|---|
| **D1. Single in-process async queue.** | All writes funnelled through one Node module's queue. | **+** Trivial implementation. **−** Specialists run in separate OS PIDs (subagent processes; agent-team panes); single in-process queue is insufficient. Rejected — the design must work across processes. |
| **D2. Lock the live log file directly.** | `flock(live_log)` on every append; `rename` on rotation. | **+** Simpler than D3. **−** Open-before-lock window: between `open(live_log)` and `flock()`, another rotator can `rename()` the file out from under us; the open fd points at the now-renamed (orphaned) inode; the next event lands in the wrong chunk. **−** Rename → recreate on rotation creates a gap where the live-log path doesn't exist. **(Drafted in round 1; rejected.)** |
| **D3. Stable lockfile sidecar (`.lock`, zero-byte, never deleted).** | All processes lock a stable inode (the `.lock` file); appender opens live log AFTER acquiring the lock; rotator uses `O_EXCL` on recreation. | **+** Lockfile inode is permanent for the run. **+** Open-after-lock eliminates the rename race. **+** `O_EXCL` on rotator-recreate defends rotator-rotator races (one wins, others retry). **−** One extra `open(.lock)` per write. |
| **D4. Per-process log files; no shared file.** | Each specialist writes its own `lane-<id>-events.jsonl`; merge step before `summary.md`. | **+** Zero race surface. **−** No single live-log to tail. **−** Merge cost on every summary-regen. **−** Cross-platform fallback for Windows still needs this anyway. |

## Decision

The chosen path is **A2 + B3 + C2 + D3** (with D4 as Windows fallback).
Concrete commitments:

### 1. Loop-skill shape — three new T2 skills wrapping existing skills

Three new skill files land under `skills/meta/` (T3b-backend-loops):

- `skills/meta/loop-clarify/SKILL.md` — the F-1 architect↔researcher
  pre-spec loop. Wraps `guild:brainstorm`. Sentinel
  `## NO MORE QUESTIONS` emitted by researcher.
- `skills/meta/loop-plan-review/SKILL.md` — the F-2
  architect↔security plan loop. Wraps `guild:plan`. Sentinel emitted
  by security.
- `skills/meta/loop-implement/SKILL.md` — the F-3 dev↔tester /
  dev↔qa / dev↔security loops with security restart. Wraps
  `guild:execute-plan`. Sentinel emitted by the per-layer challenger
  (`qa-property-based-tests`, `qa`, `security`).

Each skill conforms to the contracts pinned in
`benchmark/plans/v1.4-loop-skill-contracts.md` (machine-checkable;
verify-done greps it).

The existing `skills/meta/{brainstorm,plan,execute-plan}/SKILL.md`
files are **untouched**. The new loop skills compose the existing ones;
they do not replace them. A `--loops=none` run dispatches the existing
skills directly without involving the loop skills at all.

### 2. Plugin-surface integration — gated additions only

#### `commands/guild.md` `allowed-tools` — additive

```yaml
allowed-tools: Read, Write, Edit, Grep, Glob, Bash, Agent, Skill, AskUserQuestion, TaskCreate, TaskUpdate, TaskList
```

Four new tools appended. Existing eight unchanged. SemVer additive →
non-breaking. The TOOLS being available is distinct from them being
USED — they only invoke when `--loops=*` is set.

#### Hook events — `PreToolUse` and `PreCompact` added

Two new hook handlers under `hooks/`:

- `hooks/pre-tool-use.ts` — no-op log emitter; appends one
  `hook_event` entry to F-5 JSONL per dispatch; falls through
  gracefully when host doesn't dispatch.
- `hooks/pre-compact.ts` — same shape; logs context-preservation
  signal.

`hooks/hooks.json` updated to register both. The handler bodies are
single-shot (`appendFile` only) — p99 < 50 ms per spec Risk 9.
Existing hook registrations untouched.

#### Status-line — opt-in script

`scripts/statusline-guild.sh` outputs:
```
phase | round | cap | loops-mode | restart-count
```

Opt-in via `--statusline` CLI flag OR `GUILD_STATUSLINE=1` env var
(default off, preserves SC8). Reads `input_json` on stdin per Claude
Code statusline convention.

The script must work in BOTH single-process subagent mode AND
agent-team pane mode. Agent-team pane mode resolves a per-pane lane
identity from
`.guild/runs/<run-id>/agent-team/pane-<id>.json`, then reads the
shared `counters.json`. Mode detected by env
(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`).

#### `AskUserQuestion`/`TaskCreate` substitution at gates — gated by `--loops`

With `--loops=none`, Gates 1, 2, 3 use the v1.3 free-text path
verbatim — preserves SC8. With any other `--loops` value, gates use
native `AskUserQuestion`. `TaskCreate`/`TaskUpdate` only fire under
`--loops=*` (loop-round visibility). Falls through to free-text when
the host runtime does not dispatch `AskUserQuestion`.

### 3. JSONL schema versioning — stable repo path

Schema doc lives at the stable repo path
`benchmark/plans/v1.4-jsonl-schema.md` with `schema_version: 1` in
frontmatter. Read at runtime by the writer (`benchmark/src/log-jsonl.ts`)
and validator (`scripts/v1.4-log-validator.ts`). All 12 event types
defined per spec §"F-5 / Event types (12)" — see schema doc for the
exhaustive table.

Future v1.x additions either preserve backwards compat (additive only)
OR bump `schema_version` and ship a migration ADR. The post-rotation
summary regen contract (read live + every archive in chronological
order, deterministic output) is pinned in the schema doc.

The `.guild/runs/<id>/logs/schema.md` per-run path drafted in spec
round 1 is **rejected** (per-run runtime state cannot host stable
committed artifacts).

### 4. Stable-lockfile race control — text architecture diagram

Three writers (specialists in separate OS PIDs) and one reader (the
`summary.md` regenerator) coordinate via a stable lockfile sidecar.

```
.guild/runs/<run-id>/logs/
├── .lock                  ← zero-byte, created at run-init,
│                            NEVER deleted/renamed/truncated for the run.
│                            Inode is permanent. flock target.
├── v1.4-events.jsonl      ← live log; appendable; rotated when > 10 MiB.
└── archive/
    ├── v1.4-events.1.jsonl.gz
    ├── v1.4-events.2.jsonl.gz
    └── ...
```

#### Append path (every appender, every event)

```
appender                                lock state
--------                                ----------
1. open(.lock, O_RDONLY) → lock_fd      held by no one (yet)
2. flock(lock_fd, LOCK_EX)              ← blocks until acquired
3. open(live_log, O_WRONLY|O_APPEND|O_CREAT) → log_fd
4. write(log_fd, event_json + "\n")     ← single atomic POSIX write
5. close(log_fd)
6. flock(lock_fd, LOCK_UN)              ← released
7. close(lock_fd)
```

Step 3 happens AFTER step 2 — guarantees the open fd resolves to
whichever live log path the rotator (if any) has just left in place.
The open-before-lock window is closed.

#### Rotation path (single rotator at a time)

```
rotator                                 lock state
-------                                 ----------
1. open(.lock, O_RDONLY) → lock_fd
2. flock(lock_fd, LOCK_EX)              ← all appenders block
3. rename(live_log, archive/v1.4-events.<N>.jsonl)
4. gzip → archive/v1.4-events.<N>.jsonl.gz; rm rotated raw file
5. open(live_log, O_WRONLY|O_CREAT|O_EXCL) → recreate empty
   ↳ if EEXIST: another rotator beat us; close, goto step 1
6. close, flock(lock_fd, LOCK_UN), close(lock_fd)
```

Step 5's `O_EXCL` defends the rotator-rotator race deterministically:
the loser retries from step 1 (now sees the new live file, falls
through with no work to do, releases lock). One winner per rotation.

#### Reader path (`summary.md` regen)

```
reader                                  lock state
------                                  ----------
1. open(.lock, O_RDONLY) → lock_fd
2. flock(lock_fd, LOCK_EX)
3. read live_log into buffer            ← snapshot under lock
4. flock(lock_fd, LOCK_UN), close lock_fd
5. read archives read-only              ← immutable post-rotation
6. merge by chronological order:
   archives ascending by N → live last; ts-sort within each chunk.
```

#### Cross-platform fallback (Windows; `process.platform === "win32"`)

Per-lane files: `.guild/runs/<run-id>/logs/lane-<id>-events.jsonl`
with no shared file and no race surface. Merge step before
`summary.md` reconstructs total ordering by `ts`.

### 5. `--loops` value-set grammar

`--loops` accepts a single keyword from `none | spec | plan | implementation | all`,
OR a comma-list subset of `{spec, plan, implementation}` (e.g.
`--loops=plan,implementation`). `none` and `all` are SINGLE-keyword
forms only — neither may appear in a comma-list. Invalid values reject
with exit 2 and the literal error
`--loops must be one of none|spec|plan|implementation|all or a comma-list of {spec,plan,implementation}`.
The `GUILD_LOOPS` env mirror uses the same grammar. CLI overrides env.
Full normative description in `benchmark/plans/v1.4-claude-plugin-surface-audit.md`
§"`--loops` value set — full grammar".

### 6. Cap defaults and reset boundaries

Per spec §"Constraints" and §"Cap reset boundaries":

- **Loop cap default:** 16 rounds per loop pair. Configurable via
  `--loop-cap N` CLI / `GUILD_LOOP_CAP` env. Invalid values rejected
  exit 2. Cap range: positive integer ≤ 256.
- **Restart cap default:** 3 per lane per task (security restart
  counter). Configurable per-lane via plan override; no global env var.
- **Reset boundaries:** L1 resets at brainstorm-write-spec; L2 resets
  at user-approval-gate; L3 resets when control passes to L4 for the
  lane; L4 resets when control passes to security-review for the lane;
  security-review resets if the loop fully exits OR if security
  restarts from L3. A security restart resets L3/L4/security counters
  for that lane AND increments `restart_count` for that lane.
- **Counter persistence:** `.guild/runs/<run-id>/counters.json`
  written after every increment. Crash/resume continues from correct
  round. Run-id unique per task; reusing across tasks is undefined
  (operator mistake, out of scope).

### 7. `loops_applicable` enum + per-lane defaults

Five valid values, in fixed order:
`none`, `l3-only`, `l4-only`, `both`, `full`. Plan-validate
(T3a-backend-config) rejects others with exit 2 and the literal error
`loops_applicable must be one of: none, l3-only, l4-only, both, full`.

Defaults per lane type per spec. Security-owned lanes have NO
default — the plan must explicitly set `loops_applicable`. The T6
**review-with-pattern-patch** carve-out (T6-security maintaining
`benchmark/src/redact-log.ts` patterns) sets
`loops_applicable: none` with the literal comment marker
`# review lane; loops_applicable=none per T6 carve-out` on the same
line. Plan-validate's check is:

1. If a security-owned lane omits `loops_applicable` entirely →
   reject with exit 2 + the literal error
   `security-owned lane <lane_id> must set loops_applicable explicitly`.
2. If a security-owned lane sets `loops_applicable: none` AND the
   line carries the comment marker
   `# review lane; loops_applicable=none per T6 carve-out` → ACCEPT
   (the T6 exemption).
3. If a security-owned lane sets `loops_applicable: none` WITHOUT
   the comment marker → reject with exit 2 + the literal error
   `security-owned lane <lane_id> sets loops_applicable=none without the T6 exemption marker`.
4. If a security-owned lane sets any other value
   (`l3-only`, `l4-only`, `both`, `full`) → ACCEPT (normal path).

The comment marker is greppable; the loop-implement skill's
documentation block in `v1.4-loop-skill-contracts.md` carries the
same literal.

### 8. Termination contract — sentinel + post-sentinel regex set

Pinned in `benchmark/plans/v1.4-loop-skill-contracts.md`
verbatim. Summary:

- Sentinel: `## NO MORE QUESTIONS` (literal, on its own line).
- Post-sentinel regex set (applied to text AFTER the sentinel only):
  - `/^.*\?\s*$/m` — trailing `?` lines.
  - `/^\s*[-*]\s+(blocker|must fix|cannot proceed|MUST|BLOCKING)\b/im`
    — hard-blocker bullets.
  - `/\b(TODO|FIXME|XXX)\b/` — TODO markers (case-sensitive).
- Two consecutive malformed terminations escalate via
  `AskUserQuestion` with the standardized 3-option choice.
- The keyword set `/concern|issue|gap|missing|undefined/i` from round 2
  draft is **removed** (false positives on "no concerns remain").

## F-7 Validation Harness — design

Spec SC7 / F-7 names the validation harness's deliverables; this
section pins the design-level mechanics qa (T5) implements against.
The harness exercises every loop layer end-to-end against a
synthetic test case (`add a model_family derived field to
score.json`) and validates the run's SHAPE (presence of fields,
loop fires, log events emit, restart counter increments) — NOT the
specialist prose.

### Three injection points

The harness intentionally pre-seeds three deterministic forcings
into the synthetic-case run, each isolating one contract:

1. **Forced security-fail injection** — exercises the F-3 security
   restart path. Mechanism: the harness writes a pre-canned
   security receipt for the synthetic lane at
   `.guild/runs/<harness-run-id>/handoffs/security-<lane_id>.md`
   containing the literal pattern
   `## Findings\n- severity: high\n  addressed_by_owner: false\n  description: <synthetic blocker>`
   ABOVE the `## NO MORE QUESTIONS` sentinel (the structured-finding
   format pinned in `v1.4-loop-skill-contracts.md`
   §"Restart semantics — security restart"). The orchestrator
   detects this on first dispatch and routes to restart-from-L3 per
   the F-3 contract. Acceptance: L3 + L4 + security-review receipts
   move to `handoffs/superseded/<lane_id>-restart-1/`;
   `counters.json[<lane_id>].restart_count` increments to 1; new
   context bundle for the lane includes the finding verbatim. Test
   asserts the file moves + counter increment + bundle inclusion;
   does NOT assert lane prose.

2. **Cap-hit injection at `--loop-cap=2`** — exercises the
   cap-hit-escalate path. Mechanism: the harness invokes the run
   with `--loop-cap=2` (overrides the default 16). The synthetic
   case's L1 is pre-fixtured to require ≥ 3 rounds (the architect
   fixture intentionally returns "still have 1 question" for the
   first 2 rounds before any clean termination is possible).
   Round 3 hits the cap and triggers escalation. Acceptance:
   `escalation` JSONL event emits with `reason: "cap_hit"` and
   `options_offered: ["force-pass", "extend-cap", "rework"]`
   (the standardized escalation copy); the harness pre-seeds
   `user_choice: "force-pass"` (via stdin to the free-text
   fallback path or via fixture-injected `AskUserQuestion`
   response). Test asserts presence of the `escalation` event
   with the exact `options_offered` array; does NOT assert
   architect prose.

3. **Auto-approve injection at `--auto-approve=all`** — exercises
   the gate auto-pass + mid-execution auto-pass paths. Mechanism:
   the harness invokes the run with `--auto-approve=all`.
   Acceptance:
   - All three lifecycle gates (Gate 1 spec, Gate 2 team,
     Gate 3 plan) emit `gate_decision` JSONL events with
     `decision: "approved"` and `source: "auto-approve-mode"`.
   - Any mid-execution decision (significance ≥ medium routed via
     `guild:decisions`) emits an `assumption_logged` event with
     the assumption-text matching the decision's auto-pass
     justification, AND writes a corresponding line to
     `.guild/runs/<harness-run-id>/assumptions.md`.
   - No `AskUserQuestion` dispatch fires for gates (the auto-pass
     path skips the prompt entirely).
   Test asserts the three `gate_decision` events + the
   `assumption_logged` events + absence of `AskUserQuestion`
   tool_calls for gate dispatch.

### All-12-events-captured property

The harness's exit-condition success criterion (per spec SC7
"JSONL log captures all 12 event types at least once across the
run") is satisfied by ONE end-to-end run that exercises:

- F-1 + F-2 + F-3 (`--loops=all`) → emits `phase_start`,
  `phase_end`, `specialist_dispatch`, `specialist_receipt`,
  `loop_round_start`, `loop_round_end`, `tool_call`, `hook_event`
  (8 types).
- Auto-approve-all → emits `gate_decision`, `assumption_logged`
  (2 more, total 10).
- Cap-hit injection → emits `escalation` (11).
- Codex review at G-spec/G-plan/G-lane (active during plugin
  development, harness runs in dev context) → emits
  `codex_review_round` (12).

Test asserts: post-run, scan the JSONL log; assert each of the
12 `event` enum values appears at least once. The single run
satisfies the all-types-covered contract; no separate runs per
type are needed.

### `summary.md` regen-determinism check

After the harness completes, qa runs `scripts/v1.4-summary-md-regen.ts`
twice against the same JSONL set; assert byte-identical output
(per `v1.4-jsonl-schema.md` §"Post-rotation summary regen contract"
clause 5). This pins the determinism contract independently of the
SHAPE-not-CONTENT validation.

### Implementation owners

Per spec F-7 row of the feature decomposition table:

- **qa** — harness orchestration; SHAPE assertions; regen determinism.
- **backend** — synthetic-case implementation (the `model_family`
  derived field on `score.json`); the three injection mechanisms
  above (forced-fail receipt seeding, cap override plumbing,
  auto-approve flag handling).
- **security** — security-restart-path verification (the moved
  receipts under `handoffs/superseded/` carry the right
  `superseded_by` cross-references).
- **frontend** — frontend-lane participation in L3/L4 (the
  `model_family` synthetic field touches a frontend-rendered
  scoreboard, exercising the frontend specialist's loop layers).

### Out of scope for F-7 (deferred-with-reason)

- Property-based fuzzing of the loop converger — would
  destabilise the SHAPE-not-CONTENT contract. Re-entry trigger:
  if the synthetic case becomes flaky, expand to deterministic
  fixtures per RT5 in this ADR.
- Cross-platform validation (Windows fallback path) — the
  harness runs POSIX-only in v1.4; the Windows per-lane-files
  path is unit-tested separately in
  `benchmark/tests/log-jsonl-windows-fallback.test.ts`.

## Boundary semantics (the "additive, not supersession" subsection)

This ADR's most important contract: every v1.4 addition is **additive**
to its v1.3 counterpart. Nothing renames. Nothing replaces. The user
can run `/guild "task"` (no flags) on v1.4 and get bit-for-bit v1.3
behavior. The user can run `/guild --loops=all "task"` and the loops
fire, but the underlying skills (`guild:brainstorm`, `guild:plan`,
`guild:execute-plan`) are still the v1.3 skills.

### F-1 ↔ `guild:brainstorm` — the loop wraps, does NOT replace

`guild:loop-clarify` is a NEW skill at `skills/meta/loop-clarify/`.
It runs BEFORE `guild:brainstorm` writes the spec when `--loops=spec`
or `--loops=all` is set. Researcher's outputs feed
`guild:brainstorm`'s "Assumptions" section. With `--loops=none`, the
loop skill is never dispatched and `guild:brainstorm` runs directly,
exactly as in v1.3.

### F-2 ↔ `guild:plan` — same shape

`guild:loop-plan-review` is a NEW skill at
`skills/meta/loop-plan-review/`. It runs AFTER `guild:plan` writes
the plan AND BEFORE the user-approval Gate 3 when `--loops=plan` or
`--loops=all` is set. With `--loops=none`, the loop skill is never
dispatched and Gate 3 fires immediately on `guild:plan`'s output, as
in v1.3.

### F-3 ↔ `guild:execute-plan` — same shape, layered

`guild:loop-implement` is a NEW skill at
`skills/meta/loop-implement/`. It dispatches the per-lane L3 / L4 /
security-review layers per the lane's `loops_applicable` field. With
`--loops=none`, the layers are never dispatched and
`guild:execute-plan` runs the lane's owner specialist exactly as in
v1.3.

### v1.4-internal-loops vs dev-only-Codex-review — distinct disciplines

These are easy to conflate; they are NOT the same:

| Aspect | v1.4 internal Guild loops | dev-only Codex review |
|---|---|---|
| **Purpose** | Internal adversarial pair between Guild specialists | External second-opinion review at gates during plugin development |
| **Sentinel** | `## NO MORE QUESTIONS` | `## SATISFIED` |
| **Cap** | 16 rounds default (configurable via `--loop-cap`) | 5 rounds (separate, per `codex-adversarial-review-loop` decision) |
| **Scope** | Ships in the plugin to consumers | NEVER ships; runs only during plugin dev |
| **Surfaces at** | F-1/F-2/F-3 phases | G-spec / G-plan / G-lane gates |
| **Pair shape** | architect↔researcher (L1); architect↔security (L2); dev↔qa↔security (L3/L4/security) | Codex (external) ↔ writer (Guild specialist) |
| **Escalation copy** | `force-pass` / `extend-cap` / `rework` | (Codex discipline; backwards-compat-skip if Codex unavailable) |
| **Logged as** | `loop_round_*`, `escalation` JSONL events | `codex_review_round` JSONL event (dev-only emission; one of the 12 event types defined in `v1.4-jsonl-schema.md`) |
| **Active when** | `--loops=*` flag is set | `codex-adversarial-review` discipline is active during plugin dev |

The two are layered: when v1.4 is being DEVELOPED, BOTH disciplines run
(internal loops at runtime + Codex review at dev gates). When v1.4 is
RELEASED to consumers, only the internal loops run; consumer-side runs
never see Codex.

## Consequences

### Positive

- **Adversarial review at every gate is now structural.** Specialists
  challenge each other before output ships to the next phase. The
  user's stated v1.4 goal lands.
- **Opt-in preserves v1.3 contract.** SC8's golden fixture pins
  byte-for-byte regression-free behavior with `--loops` unset. Existing
  v1.3 consumers see zero migration cost.
- **Audit log captures every internal dispatch.** All 12 event types
  cover phase boundaries, dispatches, receipts, loop rounds, tool calls,
  hooks, gates, assumptions, escalations, and (dev-only) Codex
  rounds. Future analytics (`/guild:stats`, `/guild:evolve`) get a
  schema-stable feed.
- **Race control works across processes.** Stable-lockfile design
  closes the open-before-lock and rename-recreate-gap races. The
  `O_EXCL` retry handles rotator-rotator races deterministically. Tests
  pin: 4-parallel-process append, rotation-during-append,
  rotator-rotator, lockfile-inode persistence.
- **Plugin-surface additions are non-breaking.** All four new tools
  appended; both new hooks are no-op when host doesn't dispatch; status
  line is opt-in. No v1.3 surface removed or renamed.
- **Schema-versioning is a first-class commitment.** `schema_version: 1`
  with documented bump-on-breaking discipline; future migrations get
  ADRs (D5 + D6 satisfied).
- **Restart cap bounds the security path.** Spiral risk closed: 3
  restarts per lane per task max; on cap-hit, escalate with the
  standardized 3 options.

### Negative — accepted trade

- **Token cost grows when `--loops=all` is on.** Up to `cap × 4 layers
  × specialist dispatches` worth of model invocations per phase. This
  is an explicit user-controlled trade — operators choose by setting
  the flag. README documents cost expectations.
- **Hook overhead per host dispatch is non-zero — and is paid by every
  v1.4 install regardless of `--loops` value.** `PreToolUse` adds one
  `appendFile` per tool dispatch; `PreCompact` adds one `appendFile`
  per context-preservation event. On lanes with thousands of tool
  calls, this aggregates to real time. Bounded by spec Risk 9's
  `< 50ms p99` target; if tests show overrun, the rollback path is to
  gate the hook bodies by `GUILD_LOOPS != none` (per spec Risk 9
  rollback note line 295). The default position is always-append: the
  audit-log contract (D6) is the F-5 deliverable, and the F-5 JSONL
  path is a v1.4-new artifact whose presence is operator-opt-in by
  installing v1.4.
- **Loop logic is now 3 new skill files + 2 new hooks + 1 status-line
  script.** The plugin's surface footprint grew. Mitigation: the
  loop-skill bodies are gated by `--loops=*` (no work when unset);
  hook handlers always-append a single `hook_event` JSONL line per
  host dispatch (the constant overhead the operator opts into when
  installing v1.4 — small, bounded, audited). SC8's "byte-for-byte
  v1.3 equivalence" applies to user-visible behavior (skill outputs,
  gate prompts, free-text fallback, status-line absence) and explicitly
  excludes the F-5 JSONL path which is v1.4-new artifact.
- **Stable-lockfile race control adds one lockfile per run.** Disk
  cost is zero (zero-byte file); inode count grows by 1 per run. For
  consumers with thousands of runs, this is noise relative to the
  per-run artifact directory's size.
- **The `loops_applicable` enum is fixed at 5 values.** Adding a 6th
  requires a spec amendment. Mitigation: the 5 cover the matrix
  (L3 × L4 × security) cleanly; expanding would dilute the enum's
  testability.
- **Cross-platform fallback adds a Windows-specific code branch.**
  POSIX hosts use the stable-lockfile path; Windows uses per-lane
  files. The branch is small (one `if (process.platform === "win32")`
  at logger init) and tested by qa under both modes.

  - T3c: F-5 logger + redaction + schema validator + summary regen.

## Re-entry trigger (deferred-with-reason items)

The following items are deferred-with-reason from v1.4 to a future
release. Each names an observable end-condition:

### RT1 — Plugin manifest restructure (v2.0)

**Deferred:** rename of `hooks/agent-team/dist/*.js` paths, splitting
`commands/guild.md` into per-phase commands, re-namespacing
`skills/{core,meta,knowledge,fallback,specialists}/`. **Re-entry
trigger:** when v2.0 work begins (next major; explicit user decision)
AND a concrete operator pain point names which restructure to apply.
Without a pain point, restructure is churn.

### RT2 — `SessionEnd` hook (left on table)

**Deferred:** redundant with `Stop` for the current lifecycle.
**Re-entry trigger:** when a future use case needs a separate "session
about to terminate" signal distinct from `Stop` (e.g., flushing F-5
log buffers across cross-process writers).

### RT3 — `Notification` hook (left on table)

**Deferred:** Guild's notifications are `AskUserQuestion`-based.
**Re-entry trigger:** when a v1.x operator-experience pass adds
OS-level notifications.

### RT4 — `Monitor`, `LSP`, `EnterWorktree`/`ExitWorktree` tools

**Deferred:** not used in current lifecycle. **Re-entry trigger:** see
audit doc §6 for per-tool conditions.

### RT5 — Hard cap on F-7 synthetic cases

**Deferred:** F-7 ships one synthetic case (model_family). **Re-entry
trigger:** when consumers report flake or environmental variability
that the shape-not-content harness misses; expand the case set with
deterministic fixtures.

### RT6 — `minimumClaudeCodeVersion` pin in plugin.json

**Deferred:** evidence-conditional. **Re-entry trigger:** add the pin
**IFF AND ONLY IFF** T3d's live smoke confirms BOTH:
(a) older Claude Code versions silently skip
`PreToolUse`/`PreCompact` (the host does NOT dispatch the new hook
events), AND
(b) that silent-skip produces an operator-visible bug
(e.g. F-5 audit log missing entries an operator depends on, or
`summary.md` regen incomplete in a way operators notice).
If both conditions hold, T3d adds the pin to
`.claude-plugin/plugin.json` and updates
`docs/migration-v1.3-to-v1.4.md`. If either fails, the pin is
OMITTED — the falls-through-gracefully contract handles older hosts
and the audit log simply has fewer entries on those hosts. The
audit and ADR state the same condition; the receipt's followup
mirrors it.

### RT7 — Cross-run aggregation of F-5 logs

**Deferred:** per-run only in v1.4. **Re-entry trigger:** when
`/guild:stats` or `/guild:evolve` is built out and needs to aggregate
JSONL across runs. Today, each run is self-contained.
