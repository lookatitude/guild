# Benchmark Factory — plans index

Every plan and ADR for the `benchmark/` subsystem. Order matches phase progression: design first, then case authoring, then runner, then metrics, then forward references for P2 and P4.

| File                                  | Type         | Phase  | One-line description                                                                                                                                          |
| ------------------------------------- | ------------ | ------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `01-architecture.md`                  | Design       | P1–P4  | Multi-component design for the whole benchmark subsystem — component diagram, Case → Reflection node table, per-artifact data flow, R1–R5 failure modes, NFRs. |
| `adr-001-runner-ui-boundary.md`       | ADR (accepted) | P1   | Locks the runner ↔ UI boundary as a server JSON shape (Option A, weighted 90 vs. 71). The contract the P2 UI and the deferred public website will both consume. |
| `p2-ui-architecture.md`               | Design       | P2     | Component tree, state shape, and data-fetching pattern for the React UI shipped in P2. Extends ADR-001's six-route contract and the dotted-line `ui` subgraph in `01-architecture.md §1`. |
| `p3-runner-architecture.md`           | Design       | P3     | P3 runner subprocess + artifact-capture contract — `child_process.spawn` model, environment sanitisation, stdio capture, signal escalation, post-run capture protocol, error/retry taxonomy, and the `run.json` annotation contract. |
| `adr-003-host-repo-vs-fresh-fixture.md` | ADR (accepted) | P3 | Locks the runner cwd model as a fresh-fixture clone per run (Option A, weighted 87 vs. 57 vs. 76). Operator's host repo is never the subprocess cwd; capture is `fs.cp` of the post-run `_workspace/.guild/` tree. |
| `adr-004-runner-process-group-signaling.md` | ADR (accepted) | P3 | Amends `p3-runner-architecture.md §2.2` — flips `detached: false` → `detached: true` and signals the whole process group (`process.kill(-child.pid, SIG)`). Makes the spec's 1h cap a hard constraint instead of a soft one. |
| `security-review.md`                  | Security review | P3  | P3 runner threat model — 21 findings (9 High, 5 Medium, 7 Low) classified 16 P3-required + 5 deferred-with-reason. | `p4-learning-loop-architecture.md`    | Design       | P4     | P4 multi-component design — closes the autoresearch loop in `01-architecture.md §2`. Pins the orchestrator pattern (two-stage CLI), the `LoopManifest` and `ReflectionApplied` schemas, the keep/discard rule (`>= 2.0` AND every component `>= -1.0`) with per-case override, the optional `auth_identity_hash` field, and the three new security surfaces (file-write, manifest tampering, auth_identity_hash). |
| `adr-005-learning-loop-orchestrator.md` | ADR (accepted) | P4 | Locks Option A — two-stage CLI (`loop --start` + `loop --continue` + read-only `loop --status`; `--dry-run` on both write paths). Three options scored against six drivers: A 122 vs B 76 (interactive pause) vs C 78 (server endpoint). No new HTTP routes in v1; the loop never writes plugin source. |
