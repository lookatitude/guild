Why Guild

Skills, wiki memory, and agent superpowers working together

Guild is built around one simple constraint: the model should not improvise the operating system for your project every time it works.

01

Skills are reusable behavior

Karpathy-style skills work because they are small, explicit, and load only when useful. Guild keeps planning, review, taxonomy, evolution, and specialist craft as inspectable skills instead of hiding them in one giant prompt.

02

The wiki makes memory durable

LLM wiki principles separate raw sources from synthesized knowledge. Guild keeps facts, standards, decisions, entities, and confidence in `.guild/`, so the system can remember without losing where each claim came from.

03

Superpowers need a roster

Instead of asking one agent to be everything, Guild composes a small team for the job: architect, backend, QA, docs, researcher, marketing, or whatever the task needs. When agent teams are unavailable, the same briefs fall back to isolated subagents.

What you get

Planning discipline, specialist execution, durable memory

Question-driven planning

Guild captures goals, constraints, acceptance tests, non-goals, risk, and autonomy boundaries before execution starts.

Specialist teams

Architect, backend, QA, security, researcher, docs, SEO, marketing, sales, and more compose per task instead of all firing at once.

Context bundles

Each specialist gets a compact authoritative brief instead of a project-wide context dump.

Project memory

Raw sources, decisions, standards, product notes, concepts, and source summaries live in `.guild/` and can be committed.

Evidence gates

Review and verification consume handoff receipts with changed files, assumptions, tests, screenshots, citations, and risks.

Self-evolution

Skills and specialists evolve through reflections, shadow mode, flip reports, and versioned rollback.

How it works

The Guild system

The diagrams below map the finished plugin, but each one has a job: show what Guild owns, how it thinks, and where human approval stays in the loop.

Read them as a contract for autonomy. Guild gathers intent, selects the smallest useful team, assembles focused context, records durable memory, and only evolves skills or specialists when evidence proves the change belongs in the system.

System map

Architecture with clear ownership

This map shows the boundary between the installed plugin, the execution backends, the specialist roster, and the durable project state. It is the quickest way to see what Guild owns and what Claude Code still runs directly.

  • Commands orchestrate work without hiding state.
  • `.guild/` keeps plans, memory, runs, and evolution auditable.

Plan contract

Lifecycle

The lifecycle diagram follows a task from the first clarification questions through team composition, planning, execution, review, verification, and reflection. The important idea is that autonomy starts only after the plan has a visible contract.

  • Unknowns become questions before work begins.
  • Receipts keep review and verification grounded.

Roster selection

Team composition

This diagram explains how Guild turns an approved spec into a small, scoped team. It compares the task against available specialists, identifies coverage gaps, and asks for a deliberate choice instead of inventing authority silently.

  • Existing specialists are preferred when they fit.
  • Coverage gaps are surfaced before dispatch.

Brief assembly

Context assembly

Context assembly shows how a specialist brief is built from stable project truth, role-specific guidance, and the current task. Instead of dumping the whole repository into every agent, Guild creates a smaller brief that can be inspected and reproduced.

  • Universal context carries project invariants.
  • Task context stays focused on the current outcome.

Project memory

Durable memory

The memory diagram separates raw source material from synthesized wiki pages. That split matters because Guild can learn from the project over time while still preserving where claims came from and how confident the system should be.

  • Raw inputs stay immutable and traceable.
  • Wiki pages promote only useful, sourced knowledge.

Skill layers

Skill taxonomy

The taxonomy groups Guild's behavior into core principles, workflow skills, knowledge operations, methodology fallbacks, and specialist playbooks. This keeps the system legible when new skills are added or existing ones evolve.

  • Shared workflow skills stay separate from specialist craft.
  • Layering makes ownership and regression checks clearer.

Promotion gates

Evidence-based evolution

Evolution is intentionally conservative. Reflections can propose changes, but the diagram shows how shadow mode, comparison reports, and rollback paths keep new behavior from reaching the live roster without evidence.

  • Historical tasks test proposed behavior before promotion.
  • Regressions are archived instead of shipped.

Governed growth

New specialists earn their place

Specialist creation is the controlled path for recurring work that does not fit the current roster. Guild can propose a role, interview for boundaries, draft the specialist, test adjacent behavior, and only register it after the gates pass.

  • Boundary scans prevent duplicate or overbroad roles.
  • Failed gates return the idea to refinement without changing the live team.
