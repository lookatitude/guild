## Anomalies

### 1. researcher agent has no web tools (`tools:` = `Read, Write, Edit, Grep, Glob, Bash`)

`guild-plan.md §15.1 #12` states *"only the researcher specialist has web access by default."* The shipping agent definition (`agents/researcher.md`) does not include `WebFetch` or `WebSearch` in its `tools:` field.

**Risk:** None — this is *under*-permitting, not over-permitting. The researcher relies on the orchestrator session's web tools rather than holding them itself.

**Decision:** keep as-is, or align docs/code by either:
- adding `WebFetch, WebSearch` to `agents/researcher.md` (matches §15.1 wording), or
- editing §15.1 to clarify that web access is held by the orchestrator and surfaced to researcher via dispatch context.

Non-blocking for submission.

### 2. wiki-ingest uses `WebFetch` but isn't declared network-capable in `plugin.json`

`skills/knowledge/wiki-ingest/SKILL.md` legitimately uses `WebFetch` to ingest a URL the user hands it into the wiki. The audit-command contract anticipates a `network_capable_skills` field in `plugin.json` to whitelist this; no such field exists in the current manifest schema (Claude Code plugins don't expose it).

**Risk:** None — wiki-ingest is documented and behaves as advertised. There is just no manifest-level whitelist to compare it against.

**Decision:** keep as-is. Add a CHANGELOG note if/when Claude Code's plugin schema gains a `network_capable_skills` field.

Non-blocking for submission.

## Verdict

**PASS.** No critical or high-severity findings. Two §15.1 documentation/code-alignment items recorded; both informational. Plugin is safe to ship as `v1.0.0-beta4`.
