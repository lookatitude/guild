## Contract walk-through

Read the 4 P2 skills and commit `8628dfe` (P2 group-review fix pass). The chain after the fix is coherent end-to-end:

1. **`guild:wiki-ingest`** (`skills/knowledge/wiki-ingest/SKILL.md`) consumes a source (URL, file, pasted doc), writes the raw artifact under `.guild/raw/sources/<slug>/`, then writes a wiki page under `.guild/wiki/<category>/<slug>.md` with the full §10.1.1 base frontmatter. It accepts `type: context | standard | product | entity | concept | source` (singular) and explicitly excludes `decision` — "the `decision` type is owned by `guild:decisions`, not this skill" (line 75). Good ownership boundary.

2. **`guild:wiki-query`** (`skills/knowledge/wiki-query/SKILL.md`) searches the wiki. Its `category` filter accepts the singular `type:` enum (`context | standard | product | entity | concept | decision | source`) and maps internally to plural dirs — this matches the `type:` values that wiki-ingest and decisions actually write into frontmatter. Pre-fix the filter accepted plural dir names, which never matched the singular frontmatter; the fix in `8628dfe` (change I3) resolved that drift.

3. **`guild:wiki-lint`** (`skills/knowledge/wiki-lint/SKILL.md`) validates every page under `.guild/wiki/<category>/` against §10.1.1 base (Check #1), and pages under `.guild/wiki/decisions/` additionally against §10.3 ADR-lite additions (Check #7). Pre-fix, Check #7 was self-contradicting (said "beyond §10.1.1" but listed `confidence` as already-covered, even though `confidence` is a §10.1.1 base field). The fix in `8628dfe` (change B2) clarified that Check #7 audits §10.3 additions ONLY and that §10.1.1 base fields — including `confidence` — stay in Check #1's scope. The two checks are now decoupled.

4. **`guild:decisions`** (`skills/meta/decisions/SKILL.md`) writes ADR-lite pages under `.guild/wiki/decisions/<slug>.md`. Pre-fix its template only included the §10.3 additions (`date`, `asker`, `status`, `options`, `rationale`) and omitted the §10.1.1 base (`type`, `owner`, `source_refs`, `created_at`, `updated_at`, `sensitivity`) — so every decision page would have failed wiki-lint Check #1 on the first lint pass. The fix in `8628dfe` (change B1) extended the template to write the **union** of §10.1.1 base + §10.3 additions, with a comment explicitly naming the union so future edits don't drift.

Net chain: `wiki-ingest` writes raw+wiki with §10.1.1 base → `wiki-query` finds pages via singular `type:` filter → `wiki-lint` Check #1 enforces §10.1.1 base globally, Check #7 enforces §10.3 additions only on `decisions/` → `decisions` writes the union so both lint checks pass. No residual drift detected between producer and consumer across the 4 skills after `8628dfe`.

The group review that produced `8628dfe` (invoked via `superpowers:requesting-code-review` at the end of the P2 skill build-out) is the reason these three drift bugs (B1, B2, I3) were caught before the gate — worth repeating at every future phase boundary.

## Followups into P3+

- **Live ingest + decisions run** — first consuming repo should produce real numbers (sources ingested, decisions captured, lint findings) against the P2 §14 criterion. Track at P3 or whenever a dogfood repo is wired up.
- **Wiki-lint CI integration** — plan §10.6 says "weekly, after batches of 5+ ingests, or explicit `/guild:wiki lint`". The "after batches of 5+ ingests" trigger is documented in the skill but not yet automated (no hook or counter). Candidate for P4 hook work (auto-lint hook after N ingests).
- **Fixture runner** — `tests/wiki-lint/` has 9 scenario fixtures + README but no runner that executes them as an eval. Candidate for P3 eval-harness work (feed each fixture through `guild:wiki-lint` and compare findings against expected rule fires).
- **Raw ingest sensitivity policy** — wiki-ingest writes to `.guild/raw/sources/<slug>/`; `sensitivity:` is in the §10.1.1 frontmatter, but there's no guardrail that stops a high-sensitivity source from being written to disk or from being auto-included in context bundles. Candidate for P4/P5 hardening.
- **Decision-capture noise guard** — decisions skill has a significance threshold (low → transcript only; medium/high → persist) per §15.2, but nothing audits the volume. Consider a per-run decisions counter in the P5 `guild:reflect` skill so noisy runs surface.
- **Verification-before-completion discipline** — this gate pasted real command output verbatim per `superpowers:verification-before-completion`. Carry the same discipline forward at every phase gate; P1-audit.md already noted the lesson and P2 extends it.
