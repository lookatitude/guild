## Proposal R-001 — Missing Specialist: frontend-engineer

**Type:** new_specialist
**Threshold:** 1 occurrence (first observation; threshold for auto-queue is 3).
**Observation:** The pricing-calculator task required building and integrating
a React component tree, running Lighthouse, running axe-core, and writing
unit tests for a pure TypeScript utility. These tasks are frontend engineering
work that sits between the architect's design and the team's existing
specialists. The architect is a design-and-ADR role; the backend specialist
owns server-side concerns. Neither is the right home for "implement and test a
client component against a performance and accessibility budget."

**Proposal:** Create a `frontend-engineer` specialist:
- Scope: React/TypeScript component implementation, Storybook, accessibility
  scanning (axe-core), Lighthouse performance budgets, bundle analysis.
- Skills to author: `frontend-component`, `accessibility-audit`,
  `performance-budget`.
- Trigger signals: "implement component", "build UI", "client component",
  "Lighthouse", "axe-core", "bundle size".
- DO NOT TRIGGER for: backend API, server-side rendering logic (architect or
  backend), copy/SEO (content specialists).
- Adjacent boundary updates needed: architect (add "DO NOT TRIGGER for:
  frontend component implementation"), backend (already scoped to server-side).

**Action:** Queue for guild:create-specialist. Requires 2 more appearances in
team-compose runs before auto-queue threshold is met. Monitor.

## Proposal R-002 — Skill Improvement: context-assemble — add explicit "gap-fill" label

**Type:** skill_improvement
**Target skill:** guild:context-assemble
**Observation:** When a specialist role in the team.yaml does not match a
roster entry (e.g., `frontend-engineer` in this run), the context bundle was
assembled with a role name that doesn't resolve to an agent definition. The
skill produced a valid bundle but did not flag the gap-fill status in the
bundle's frontmatter. Downstream skills (review, verify) had to infer this
from the plan's assumption notes.

**Proposal:** Add a `gap_fill: true` frontmatter field to context bundles when
the role does not match a roster specialist. This gives review and verify-done
a machine-readable signal to note "this lane was run without a dedicated
specialist agent definition; quality bar may differ."

**Evidence for improvement:** In this run, the gap was surfaced via a manual
assumption note (A-001) rather than a structured field. Automated detection
would reduce orchestrator cognitive load.

**Priority:** medium.

## Proposal R-003 — Skill Improvement: team-compose — prompt for legal review gate on pricing tasks

**Type:** skill_improvement
**Target skill:** guild:team-compose
**Observation:** Assumptions A-007 (trademark check) and A-008 (legal
disclaimer sign-off) both arose because the team-compose stage did not include
a legal/compliance nudge when the task involves pricing copy. The spec's
autonomy_policy did call for legal review, but team-compose did not surface a
prompt to confirm that a legal review gate was in place before dispatching
copywriter.

**Proposal:** Add a heuristic in team-compose: if the spec contains
"pricing", "cost", "payment", or "billing" in its goal or scope, and a
copywriter specialist is included, emit a planning note:
"This task involves pricing copy. Confirm a legal review gate is in the
deployment process before proceeding."

**Priority:** low (the spec's autonomy_policy already covers this; proposal is
a belt-and-suspenders nudge).

## Proposal R-004 — Process: establish a "[product]-placeholder" lint rule

**Type:** process
**Observation:** Both fe-001 and seo-001 used "[product]" as a placeholder
throughout their outputs (component copy, title tag, meta description, FAQ
schema). These placeholders must be replaced before any production deploy. The
verify-done stage flagged this, but there is no automated check.

**Proposal:** Add a verify-done sub-check: `grep -r '\[product\]'
<changed_files>` → must return 0 matches before the run is marked production-
ready. This is a lightweight lint rule that could be added to the
guild:verify-done skill or as a hook check.

**Priority:** medium — prevents accidental "[product]" string shipping to
production.
